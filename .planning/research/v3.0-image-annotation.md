# v3.0 Research — Image & Diagram Annotation

## Executive Summary

For SafeStart v3.0's SOP Builder, the recommended annotation library is **Konva + react-konva** (MIT, ~55 KB min+gzip for core Konva, plus react-konva wrapper). It is the only MIT-licensed, actively maintained, React-bound canvas library that:

1. Ships native `Konva.Arrow`, `Konva.Rect`, `Konva.Ellipse`, `Konva.Text`, `Konva.Label`, and `Konva.Line` (freehand) — every primitive in the v3.0 requirement list.
2. Serializes to JSON via `toJSON()` and reconstructs via `Konva.Node.create()`, enabling non-destructive re-editing.
3. Has first-party React bindings (`react-konva`) with stable Next.js App Router support when loaded via `next/dynamic({ ssr: false })`.
4. Supports pointer events (stylus/Apple Pencil) via its built-in touch/pointer event system.
5. Allows bundle isolation — admin authoring pages import the editor dynamically; worker read pages never pull it.

**tldraw is now effectively off-limits**: the SDK 4.0 license change (September 2025) makes production use conditional on a commercial license (~USD $6,000/team/year) or a non-commercial hobby license with a mandatory "Made with tldraw" watermark. A paid B2B SaaS targeting NZ industrial orgs cannot ship tldraw without that annual fee.

**Excalidraw is a viable fallback** (MIT, React-first, has a native "Annotate image" feature), but its hand-drawn aesthetic, heavier bundle, and whiteboard-centric UX are a poor fit for precise industrial SOP diagrams where workers need unambiguous visual instructions.

**Storage format**: Konva JSON scene graph stored in a new `sop_image_annotations` table, with image URLs kept out of the scene and rehydrated at render time. On publish, bake a composite PNG to Supabase Storage for the mobile read path (workers never load Konva). Dual-store gives us both re-editability and fast mobile rendering.

## Storage Format Recommendation

### Decision: Konva JSON scene graph in Postgres, images referenced by URL

**Why not SVG string?**
- SVG is verbose and hard to mutate programmatically.
- Re-opening an SVG blob in an editor means re-parsing DOM, not hydrating typed objects.
- Bendable arrows with control points are awkward in SVG path syntax.

**Why not W3C Web Annotation Data Model?**
- Designed for "region references metadata" semantics (annotate a painting with a caption), not for "draw an arrow with a specific color/width pointing here."
- Mapping arrows, callouts, and text decorations into WADM requires custom selectors anyway — we lose the standard's benefit.
- No tooling advantage for our use case.

**Why not Yjs / CRDT?**
- v3.0 is single-author editing. Collaborative editing is not a requirement.
- Yjs adds ~60 KB and complexity we do not need. Defer until a future "multiple admins editing simultaneously" milestone if ever.

**Recommended schema**:

```sql
create table sop_image_annotations (
  id uuid primary key default gen_random_uuid(),
  sop_image_id uuid not null references sop_images(id) on delete cascade,
  organization_id uuid not null references organizations(id),
  scene jsonb not null,             -- Konva stage JSON (shapes only, no image refs)
  scene_version smallint not null default 1,
  natural_width int not null,       -- authoring canvas size in CSS px
  natural_height int not null,
  baked_storage_path text,          -- Supabase Storage path to flattened PNG
  baked_at timestamptz,
  created_by uuid references auth.users(id),
  updated_by uuid references auth.users(id),
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
create index on sop_image_annotations(sop_image_id);
```

**Scene JSON sketch** (stored in `scene`):

```json
{
  "schemaVersion": 1,
  "width": 1600,
  "height": 1200,
  "shapes": [
    {
      "id": "a1",
      "type": "Arrow",
      "points": [120, 80, 420, 310],
      "stroke": "#FACC15",
      "strokeWidth": 6,
      "pointerLength": 18,
      "pointerWidth": 18,
      "tension": 0.3
    },
    {
      "id": "c1",
      "type": "Label",
      "x": 440,
      "y": 290,
      "tag": { "fill": "#FACC15", "cornerRadius": 12, "pointerDirection": "left", "pointerWidth": 10, "pointerHeight": 10 },
      "text": { "text": "1  Pressure gauge", "fontSize": 20, "fontFamily": "Inter", "fill": "#111", "padding": 10 }
    },
    {
      "id": "r1",
      "type": "Rect",
      "x": 680, "y": 420, "width": 260, "height": 180,
      "stroke": "#FACC15", "strokeWidth": 4, "dash": [12, 6]
    }
  ]
}
```

Storing in `jsonb` gives us:
- Postgres-level querying (find all SOPs with more than N annotations, etc.)
- Easy schema migration via `schemaVersion` field.
- Dexie offline mirror is just a straight copy of the same object.

## Mobile Read Path

### Decision: Dual-store — JSON scene for editing, baked PNG for worker rendering

**Rationale**:

| Strategy | Worker bundle impact | First render time | Editable | Verdict |
|---|---|---|---|---|
| Live-render via Konva on mobile | +85 KB JS + canvas init | ~200–400 ms on 5-year-old Android | Yes | Penalizes every worker for an authoring feature. **Reject.** |
| Bake to PNG on publish only | 0 KB extra on workers | ~native `<img>` speed | No (re-edit requires JSON too) | Lose re-editability. **Reject alone.** |
| **Dual-store: JSON + baked PNG** | **0 KB extra on workers** | **Native `<img>` speed** | **Yes (admin reloads JSON in editor)** | **Best of both.** |

**Implementation**:

1. Admin draws annotations in the Builder. Save writes `sop_image_annotations.scene` JSON only (fast).
2. On SOP publish (`status: draft → published`), a server action (or background job) executes the bake step:
   - Headless rehydrate: load the original image and the scene JSON into a node-canvas or Konva-in-node stage.
   - Alternatively (simpler): use a small Supabase Edge Function / Next.js API route that runs `konva-node` + the scene JSON, exports to PNG via `stage.toDataURL()`, and uploads to `sop-images/baked/{sop_id}/{image_id}.png`.
3. Write the result path to `sop_image_annotations.baked_storage_path` and `baked_at`.
4. Worker read path in `(protected)/sops/[sopId]/walkthrough` loads `baked_storage_path` if present, else falls back to `storage_path` (the raw image). Never imports Konva.
5. When an admin re-opens the image for editing, the Builder loads both `storage_path` (as background layer) and `scene` (as the shape layer), not the baked file. Non-destructive.

**Bake implementation options** (pick during executor phase):
- **Option A — Client-side bake**: Admin browser runs `stage.toDataURL()` then uploads the blob. Simplest, no server dependency, but requires the browser to be open at publish time. Good for v3.0.
- **Option B — Server-side bake**: Next.js API route using `konva-node` or Playwright-driven screenshot. More robust, runs on publish regardless of admin presence. Defer to v3.1 if Option A proves flaky.

Start with **Option A** — the admin is publishing from the Builder anyway, so they're already on the page with the hydrated stage.

## Integration with `sop_images`

Current shape (from `src/types/sop.ts` and migrations):

```
sop_images
  id uuid
  sop_id / section_id / step_id
  storage_path text
  ...
```

**Non-breaking extension strategy**:

1. **Do not modify `sop_images`** for the annotation layer. Create a new `sop_image_annotations` table (see schema above) with a FK on `sop_image_id`.
2. Worker render path continues to read `sop_images.storage_path`. If an annotation row exists AND `baked_storage_path` is populated, the worker read path prefers the baked path. A single `LEFT JOIN` on `sop_image_annotations` (or an RPC view) is enough.
3. For `useSopSync` / Dexie mirror: add a new Dexie table `sopImageAnnotations` that mirrors the same structure. The existing `sop_images` offline sync stays untouched.
4. RLS: copy the existing organization-scoped policies from `sop_images` to `sop_image_annotations`. Admins write, all org members read.
5. Backward compatibility: SOPs created before v3.0 have zero rows in `sop_image_annotations` — worker path falls through to the raw image as it does today. Zero regression risk for v1/v2 SOPs.

**One new RPC** to simplify the join on the read side:

```sql
create or replace function sop_image_with_annotation(p_image_id uuid)
returns table (
  id uuid,
  storage_path text,
  display_path text,   -- baked_storage_path if present, else storage_path
  has_annotations bool
) language sql stable as $$
  select si.id,
         si.storage_path,
         coalesce(sia.baked_storage_path, si.storage_path) as display_path,
         (sia.id is not null) as has_annotations
  from sop_images si
  left join sop_image_annotations sia on sia.sop_image_id = si.id
  where si.id = p_image_id;
$$;
```

## Top Pick

**Library**: **Konva + react-konva** (MIT)

**Why**:
- All required primitives ship natively: `Arrow` (with arrowhead + bendable via `tension` + control points), `Rect`, `Ellipse`, `Text`, `Label` (tag+text group = numbered callout), `Line` (freehand).
- MIT licensed — no per-seat fees, no watermark, no production restrictions. Critical for a B2B SaaS.
- Official React bindings (`react-konva`), actively maintained, used in production by Label Studio and many other annotation tools.
- Clean JSON serialization (`stage.toJSON()` / `Konva.Node.create()`) → maps directly to our non-destructive storage requirement.
- First-class pointer event support → Apple Pencil works without shims.
- Tree-shakable core via `react-konva/lib/ReactKonvaCore` if bundle pressure grows.
- Bundle stays on admin authoring pages only via `next/dynamic({ ssr: false })`; worker pages see zero bytes of Konva because we serve baked PNGs.
- Scene JSON is plain data — Dexie mirror and Supabase sync are trivial.

**Risks**:
- **Next.js 16 `canvas` module resolution**: react-konva has a long-running issue (#832) where `Module not found: Can't resolve 'canvas'` appears in Next 13+. Mitigation: add `serverExternalPackages: ['canvas']` to `next.config.ts`, and use `dynamic(() => import('...'), { ssr: false })`. Verified working in Next 15; should hold for Next 16 but needs a spike in the first executor phase.
- **Transformer UI is DIY**: Unlike Fabric, Konva doesn't auto-draw resize handles. `Konva.Transformer` exists but the shape selection / multi-select / delete / undo-redo wiring is hand-rolled. Budget ~1 day for basic transformer UX.
- **Text editing**: Konva has no inline text editing. Standard pattern is to overlay a `<textarea>` absolutely positioned over the Konva text node while editing, then commit back. ~0.5 days work.
- **Arrow bendability**: `Konva.Arrow` is a straight line by default. For bendable arrows, pass multiple points + `tension: 0.3` to render a curved spline through control points. We'd build a control-point editor on top. ~1 day.
- **`toJSON` does not persist images/filters/listeners**: Non-issue for our storage model — images are referenced by URL outside the scene and hydrated at load time; filters/listeners are applied imperatively after hydration.

**Fallback**: **Excalidraw** (MIT), imported via `next/dynamic`. Ships with native "Annotate image" + arrows/shapes/text/freedraw out of the box. Use if (a) Konva hits a Next 16 blocker we can't work around in a day, or (b) timeline forces a zero-custom-UI solution. Accept the hand-drawn aesthetic as a tradeoff.

**Second fallback**: **Custom React SVG overlay**. Viable for rect/ellipse/arrow/text-only (drop freehand). Worth about 1 week of engineering to reach Konva feature parity for our specific primitives. Only pursue if both Konva and Excalidraw fail.

## Pitfalls to Avoid

### 1. EXIF orientation / HEIC double-rotation

iPhone HEIC images store rotation in **both** EXIF `Orientation` **and** HEIF `irot` — some decoders apply one, some apply both, producing upside-down or rotated-90 images depending on the consumer.

**Consequences**: Annotation coordinates drawn on an image as displayed in Chrome (which auto-rotates EXIF) will be wrong when the same image is read by OpenCV, server-side Sharp, or any tool that does not honor EXIF.

**Mitigation**:
- In the Phase 5 upload pipeline, **normalize every uploaded image** to JPEG or PNG with EXIF orientation **physically applied** and stripped. Store only the pixel-correct, rotation-baked image.
- Use `sharp().rotate()` in the ingestion API route — it honors EXIF and flattens the rotation into pixels.
- Never rely on browser auto-rotation for annotation math.
- Reject HEIC at the upload boundary: transcode to JPEG via `heic-convert` or OpenAI Vision input normalization before storing.

### 2. Coordinate drift on resize

Annotation coordinates are in **pixels**. If the authoring canvas is 1600x1200 and the worker views it at 390x293 (scaled for iPhone SE), arrow endpoints will land in the wrong place unless coordinates are normalized.

**Mitigation**:
- Store the authoring canvas dimensions explicitly in `sop_image_annotations.natural_width` / `natural_height`.
- Either (a) render the baked PNG at `natural_width x natural_height` and let the browser scale the whole image + annotations together (**preferred — baked PNG solves this automatically**), or (b) normalize scene coordinates to 0–1 float space and multiply by display size.
- Going with the baked PNG read path eliminates this problem entirely for workers. Only the admin editor needs coordinate logic, and it always renders at `natural_width x natural_height` in a scroll container.

### 3. Color space / HEIC → sRGB

HEIC can carry Display-P3 color profiles. Converting with a naive decoder drops to sRGB without tone mapping, making safety-critical colors (red/yellow warnings) look dull.

**Mitigation**:
- Always pass through Sharp with `.toColorspace('srgb')` explicitly and `.jpeg({ quality: 92 })` for the normalized output.
- Test on an iPhone 15 photo of a yellow forklift — confirm the annotation yellow (#FACC15) matches the photo after ingest.

### 4. Canvas memory leaks in React StrictMode

Konva `Stage` instances held in refs are NOT garbage collected if the component remounts (which StrictMode causes in dev). Memory grows unbounded.

**Mitigation**:
- Always call `stage.destroy()` in `useEffect` cleanup.
- Wrap Stage creation in `useRef` + `useEffect` idempotent init, not in render body.
- Test in dev with React StrictMode enabled — if the stage doubles up, cleanup is missing.

### 5. Stylus palm rejection on iPad

Default browser pointer events fire for both pen and palm contact, causing stray marks when the author rests their hand on the screen.

**Mitigation**:
- Filter pointer events by `pointerType === 'pen'` when a pen is detected; fall back to treating touch as pen if no pen has been seen.
- Add a UI toggle "Pen only" in the Builder toolbar for iPad users.
- Konva exposes `e.evt.pointerType` in its event objects — straightforward to filter.

### 6. Next.js App Router `canvas` module resolution

react-konva in Next.js 13+ repeatedly hits `Module not found: Can't resolve 'canvas'` because Konva's Node target pulls a dep that shouldn't be bundled for the browser.

**Mitigation**:
- `next.config.ts`: `serverExternalPackages: ['canvas']` (Next.js 15+ stable; was `experimental.serverComponentsExternalPackages` pre-15).
- Always import the editor via `const AnnotationEditor = dynamic(() => import('./AnnotationEditor'), { ssr: false })`.
- The editor component itself must start with `'use client'`.
- Spike this on day 1 of the v3.0 executor phase — do not assume it Just Works.

### 7. Offline authoring sync

Authors on construction sites may lose connectivity mid-edit. The `scene` JSON must merge cleanly when they reconnect.

**Mitigation**:
- Store pending annotation edits in Dexie keyed by `sop_image_id` + `updated_at`.
- On reconnect, use last-write-wins per image (simple, correct for single-author v3.0).
- If two admins edited the same image offline, server takes the newer `updated_at` and queues a "merge conflict" notification in the Builder — defer auto-merge to post-v3.0 since it requires CRDT or operational transforms.

### 8. `toJSON()` dropping unserializable data

`Konva.Stage.toJSON()` omits images, filters, and event listeners. If you load a scene expecting an image node to exist, it won't.

**Mitigation**:
- Never put background image URLs inside the scene JSON. Store them in `sop_images.storage_path` and add them as a background Layer imperatively during hydration, below the annotation Layer.
- Treat the scene JSON as purely shapes-over-transparent-background.
- Test round-trip serialization early: draw → save → reload → confirm every shape returns pixel-identical.

### 9. Bundle leak to worker pages

If any shared component (e.g., a Layout, Nav, or Context Provider) ever statically imports the annotation editor, Next.js tree-shaking fails and workers download 100+ KB of Konva for no reason.

**Mitigation**:
- Annotation editor lives under `src/components/admin/sop-builder/` and is imported ONLY from `(protected)/admin/sops/[sopId]/review` and `(protected)/admin/sops/[sopId]/builder` routes.
- Verify with `next build` output — the worker walkthrough route's First Load JS should not grow when the annotation editor is added.
- Add a CI check or a manual `ANALYZE=true npm run build` review before merging v3.0.

### 10. Supabase Storage cache-control on baked PNGs

Baked annotation PNGs will be regenerated on every SOP republish. If the CDN caches aggressively with the same path, workers see stale annotations.

**Mitigation**:
- Either (a) write baked files with a content-hash or version suffix: `baked/{sop_id}/{image_id}.v{N}.png` and bump `N` on republish, or (b) set short `Cache-Control` on baked files and rely on ETag.
- Prefer (a) — simpler, works with aggressive CDN caching. Update `baked_storage_path` in the DB to point at the new version; old ones garbage-collected nightly.
