# v3.0 Research — Extensible Section Schema + Reusable Blocks

## Executive Summary

**Good news: v2.0 is already further along than the milestone brief assumed.** The current schema (`supabase/migrations/00003_sop_schema.sql`) defines `sop_sections.section_type` as plain `text not null`, **not** a Postgres enum. There is no enum constraint to drop. The rendering layer (`SopSectionTabs.tsx`, `SectionContent.tsx`) already uses substring matching (`type.includes('hazard')`) and a `SECTION_DISPLAY_NAMES[x] ?? toTitleCase(x)` fallback, so arbitrary custom section types already render without crashing.

**That means v3.0 is not a destructive schema change — it's an additive layer:**

1. Introduce a `section_kinds` catalog table that categorises free-text types into rendering families (hazard, ppe, steps, content, signoff, custom) plus render metadata (icon, colour, priority). Keep `sop_sections.section_type` as free text; add an **advisory FK** `section_kind_id` that resolves to a row in the catalog. Unknown types fall back to the `custom` kind.
2. Add a block library as two tables: `blocks` (definitions, versioned, per-org or global) and `sop_section_blocks` (junction pinning a section to a block version). Default to **pin-on-add**, with a "Check for updates" affordance.
3. Handle global (null `organisation_id`) vs per-org via a single `blocks` table with a nullable `organisation_id` and an RLS policy that unions the two.
4. Expose all of this through a new wizard step that queries the block library filtered by `kind` and SOP category.

**Overall recommendation: Option A (section_kinds catalog) + single `blocks` table with pin-on-add versioning.** This avoids any migration that rewrites existing `section_type` values, preserves RLS simplicity, and slots cleanly into the existing `SopSection` TypeScript type with an optional `kind` join.

## Schema Options Comparison

### Option A — Free-text `section_type` + `section_kinds` catalog (advisory FK) — RECOMMENDED

```sql
-- Catalog of known kinds with rendering metadata
create table public.section_kinds (
  id               uuid primary key default gen_random_uuid(),
  organisation_id  uuid references public.organisations(id) on delete cascade,
  -- NULL organisation_id = global/canonical kind (hazard, ppe, steps, emergency, signoff, content, custom)
  slug             text not null,              -- 'hazard', 'ppe', 'steps', 'preflight_checks', etc.
  label            text not null,              -- display label
  render_family    text not null check (render_family in
                     ('hazard','ppe','steps','content','signoff','emergency','custom')),
  icon             text,                       -- lucide icon name
  color_token      text,                       -- 'red-400', 'blue-400', etc.
  render_priority  int not null default 100,   -- lower = earlier in walkthrough
  description      text,
  created_at       timestamptz not null default now(),
  updated_at       timestamptz not null default now(),
  unique (organisation_id, slug)              -- allow same slug per org; globals keyed by (null, slug)
);

-- Advisory FK on sop_sections (nullable; sections without a kind fall back to render_family='custom')
alter table public.sop_sections
  add column section_kind_id uuid references public.section_kinds(id) on delete set null;

create index idx_sop_sections_kind on public.sop_sections (section_kind_id);
```

**Backward compat:** Existing `sop_sections` rows keep their free-text `section_type` untouched. A backfill job walks each row and sets `section_kind_id` by matching `section_type` against canonical slugs (with the same substring logic the UI already uses). Rows that don't match remain `section_kind_id = null` and render via the `custom` fallback — which is exactly today's behaviour.

**Worker walkthrough compatibility:** The walkthrough reads `render_family` + `render_priority` + `icon` + `color_token` from the joined `section_kinds` row. If `section_kind_id` is null, it falls back to the current substring-matching logic. **Zero regression risk.**

**Pros:**
- Zero breaking migration — `section_type` stays as-is.
- Canonical vs custom is one column (`organisation_id IS NULL`).
- Render metadata is data, not hard-coded in React. Adding a new canonical kind = one SQL insert.
- Admin can create org-specific kinds (e.g. "Pre-flight checks") without DB migrations.
- Wizard queries are trivial: `select * from section_kinds where organisation_id is null or organisation_id = :org`.

**Cons:**
- Advisory FK means you can have rows where `section_type` and `section_kind_id` disagree (e.g. text says "hazard" but FK points at "ppe"). Mitigate with a trigger on insert/update that auto-links if slug matches.
- Two sources of truth for "what kind is this" — `section_type` (free text, legacy) and `section_kind_id`. Long-term we should deprecate `section_type` in favour of `section_kind_id`, but not in v3.0.

### Option B — Drop enum, make `type` free-text, add `section_templates` table

Not applicable — **there is no enum to drop.** `section_type` is already free text. This option would only be meaningful if we were starting from a constrained schema. Skipping.

### Option C — Add `custom_type_label` alongside enum

Same story as Option B. No enum exists. This option would introduce a second text column for no benefit.

### Option D — Polymorphic JSONB `section_schema_id` with type descriptor

```sql
alter table public.sop_sections
  add column schema_descriptor jsonb;
-- schema_descriptor = { "kind": "preflight_checks", "fields": [...], "render": {...} }
```

**Pros:** maximally flexible; admins could define sections with arbitrary custom fields (e.g. "Checklist with inspector signature box").
**Cons:**
- JSONB is opaque to RLS, harder to index, harder to validate.
- Zod schemas for arbitrary user-defined shapes become a runtime concern, not a design-time one.
- Worker walkthrough rendering becomes a mini template engine.
- Over-engineered for v3.0's actual requirement: **workers see the section; admins pick from a block library**. We don't need per-field schema customisation yet.

**Verdict:** Defer to v3.1 or later if we ever need fully user-defined field shapes. Not needed for v3.0.

## Reusable Blocks — Recommended Data Model

**Single `blocks` table with typed content JSONB + `block_versions` for history + junction table for SOP usage.**

```sql
-- 1. Block definitions (the "library")
create table public.blocks (
  id               uuid primary key default gen_random_uuid(),
  organisation_id  uuid references public.organisations(id) on delete cascade,
  -- NULL = global block (WorkSafe NZ standards, etc.)
  kind_slug        text not null,            -- 'hazard', 'ppe', 'step', 'emergency', etc.
                                             -- matches section_kinds.slug for wizard filtering
  category         text,                     -- SOP category filter (e.g. 'glass_manufacturing', 'hot_work')
  title            text not null,            -- 'Hot surface >200°C'
  search_tokens    tsvector generated always as
                     (to_tsvector('english', coalesce(title,'') || ' ' || coalesce(description,''))) stored,
  description      text,
  current_version  int not null default 1,
  archived         boolean not null default false,
  created_by       uuid references auth.users(id),
  created_at       timestamptz not null default now(),
  updated_at       timestamptz not null default now()
);

create index idx_blocks_org_kind on public.blocks (organisation_id, kind_slug) where archived = false;
create index idx_blocks_search on public.blocks using gin (search_tokens);

-- 2. Block versions (history; each edit bumps a new version row)
create table public.block_versions (
  id               uuid primary key default gen_random_uuid(),
  block_id         uuid not null references public.blocks(id) on delete cascade,
  version          int not null,
  content          jsonb not null,           -- { "text": "...", "severity": "critical", "ppe_items": [...] }
  change_note      text,
  created_by       uuid references auth.users(id),
  created_at       timestamptz not null default now(),
  unique (block_id, version)
);

create index idx_block_versions_block on public.block_versions (block_id, version desc);

-- 3. Junction: which blocks a section uses, at which version
create table public.sop_section_blocks (
  id                 uuid primary key default gen_random_uuid(),
  section_id         uuid not null references public.sop_sections(id) on delete cascade,
  block_id           uuid not null references public.blocks(id) on delete restrict,
  pinned_version     int not null,              -- pin-on-add default
  follow_latest      boolean not null default false, -- opt-in floating reference
  sort_order         int not null default 0,
  snapshot_content   jsonb,                     -- cached copy of pinned version at link time
  overridden         boolean not null default false, -- admin edited the snapshot away from the block
  created_at         timestamptz not null default now()
);

create index idx_ssb_section on public.sop_section_blocks (section_id, sort_order);
create index idx_ssb_block on public.sop_section_blocks (block_id);
```

### Why these choices

**Single `blocks` table + `kind_slug` discriminator** (not separate `hazard_blocks` / `ppe_blocks`):
- Matches the extensible section kinds design — adding a new kind doesn't require a new table.
- `content JSONB` on versions stays small (a hazard block is just a few fields). Zod schema per kind validates shape at the application layer.
- One set of RLS policies covers everything.

**`block_versions` table (not just a `version` column on `blocks`)**:
- Full audit trail — critical for safety-regulated environments.
- Answers "which SOPs referenced v2 of this block before we pushed v3?"
- Supports compare/diff UI.

**`sop_section_blocks` junction with `pinned_version` + `follow_latest`**:
- Default is pin-on-add. When admin inserts "Hot surface >200°C", we write `pinned_version = blocks.current_version` and cache `snapshot_content`.
- If admin ticks "Always use latest", `follow_latest = true` and the walkthrough resolves via `current_version` at query time.
- `snapshot_content` means workers can render offline without a second join — the Dexie cache already serialises the section, so it just includes the snapshot.
- `overridden = true` means an admin edited the content after inserting — the block reference stays for "where did this come from" tracking, but edits don't propagate back.

**Update propagation flow (global block → 30 SOPs):**
1. Admin edits "Hot surface >200°C" to "Hot surface >180°C" — we insert a new `block_versions` row (version 2) and bump `blocks.current_version`.
2. A background job (or deferred RPC call on edit) scans `sop_section_blocks` for that `block_id`:
   - `follow_latest = true` → snapshot_content refreshed immediately, SOP `updated_at` bumped, workers pick up on next sync.
   - `follow_latest = false` → flag shown to admin: "3 blocks in this SOP have updates available. Review." Admin clicks → snapshot_content refreshed + `pinned_version` bumped.
3. Workers only ever read `sop_section_blocks.snapshot_content`, so offline-first behaviour is unchanged.

**This matches the transclusion vs substitution pattern from CMS literature:** pin-on-add is substitution (safer default for safety-critical content), follow-latest is transclusion (faster rollout of policy updates).

## Org-scoping Strategy

**Single table, nullable `organisation_id`, RLS policy unions.** Simpler than separate `global_blocks` + `org_blocks` tables, no materialized view needed.

### RLS policy pseudocode

```sql
alter table public.blocks enable row level security;
alter table public.block_versions enable row level security;
alter table public.sop_section_blocks enable row level security;

-- Read: members see their own org's blocks + global blocks
create policy "blocks_read_org_plus_global"
  on public.blocks for select to authenticated
  using (
    organisation_id is null
    or organisation_id = public.current_organisation_id()
  );

-- Write: admins can only mutate their own org's blocks, never globals
create policy "blocks_write_own_org_only"
  on public.blocks for insert to authenticated
  with check (
    organisation_id = public.current_organisation_id()
    and public.current_user_role() in ('admin', 'safety_manager')
  );

create policy "blocks_update_own_org_only"
  on public.blocks for update to authenticated
  using (
    organisation_id = public.current_organisation_id()
    and public.current_user_role() in ('admin', 'safety_manager')
  )
  with check (
    organisation_id = public.current_organisation_id()
  );

create policy "blocks_delete_own_org_only"
  on public.blocks for delete to authenticated
  using (
    organisation_id = public.current_organisation_id()
    and public.current_user_role() in ('admin', 'safety_manager')
  );

-- Global blocks are maintained by service_role only (admin client)
-- No authenticated policy allows organisation_id IS NULL writes → cross-org contamination is impossible at the DB level.

-- block_versions inherits org scoping via join to blocks
create policy "block_versions_read"
  on public.block_versions for select to authenticated
  using (exists (
    select 1 from public.blocks b
    where b.id = block_versions.block_id
      and (b.organisation_id is null or b.organisation_id = public.current_organisation_id())
  ));

create policy "block_versions_write_own_org"
  on public.block_versions for insert to authenticated
  with check (exists (
    select 1 from public.blocks b
    where b.id = block_versions.block_id
      and b.organisation_id = public.current_organisation_id()
      and public.current_user_role() in ('admin', 'safety_manager')
  ));

-- sop_section_blocks scoped via join through sop_sections → sops
create policy "ssb_read_org"
  on public.sop_section_blocks for select to authenticated
  using (exists (
    select 1 from public.sop_sections sec
    join public.sops sop on sop.id = sec.sop_id
    where sec.id = sop_section_blocks.section_id
      and sop.organisation_id = public.current_organisation_id()
  ));

-- Same pattern for insert/update/delete — admin role required.
```

**Key safety property:** No `authenticated` role policy permits inserting or updating rows where `organisation_id IS NULL`. Global blocks are seeded via migration or service_role calls only. This prevents any admin from accidentally (or maliciously) promoting an org-specific block to global, or editing another org's data.

**Performance:** Index on `(organisation_id, kind_slug)` with a partial `where archived = false` covers the hot-path wizard query. `organisation_id is null OR organisation_id = X` is union-friendly for the planner when paired with an index.

**Alternative considered: materialized view per org.** Rejected — materialised views are maintenance burden, don't auto-refresh cleanly when globals change, and Supabase RLS on MVs is fiddly. The nullable-column approach with a single index is simpler and performant at the scale of SafeStart (hundreds of blocks per org, not millions).

## Block Versioning — Recommended Approach

**Pin-on-add by default, opt-in floating reference, explicit update review for pinned SOPs.**

| Strategy | SOP behaviour when block changes | When to use |
|----------|----------------------------------|-------------|
| **Pin-on-add (default)** | SOP keeps old content. Admin sees "updates available" badge. | Safety-critical content. Regulatory SOPs. Most blocks. |
| **Follow-latest (opt-in)** | SOP auto-refreshes snapshot_content on next block edit. | Evergreen references like company-wide PPE policies where the admin trusts central updates. |
| **Override (after pin)** | SOP edits snapshot_content locally, `overridden = true` flag set. Block reference preserved for lineage but updates are ignored. | When a specific SOP needs a variant ("PPE + hearing protection for this specific machine"). |

### Update propagation pseudocode

```typescript
// Called from server action after block_versions insert
async function propagateBlockUpdate(blockId: string, newVersion: number, newContent: Json) {
  const supabase = createAdminClient()

  // 1. Auto-refresh follow_latest subscribers
  await supabase
    .from('sop_section_blocks')
    .update({ snapshot_content: newContent, pinned_version: newVersion })
    .eq('block_id', blockId)
    .eq('follow_latest', true)
    .eq('overridden', false)

  // 2. Flag pinned subscribers for review (write to a notifications table or a pending_updates column)
  await supabase
    .from('sop_section_blocks')
    .update({ update_available: true })
    .eq('block_id', blockId)
    .eq('follow_latest', false)
    .eq('overridden', false)
    .lt('pinned_version', newVersion)
}
```

Add `update_available boolean not null default false` to `sop_section_blocks` to surface the review state without an extra join.

### Admin UX for updates

- "Block Library" page shows each block with a count of "linked SOPs: 30 (28 up-to-date, 2 pinned to older)".
- SOP review page shows a banner: "3 blocks have updates. [Review changes]".
- Each block shows a diff (old snapshot vs new version content) — admin clicks "Apply" per block or "Apply all".
- Applying fires a version-level diff update and bumps the SOP's `updated_at` so offline workers resync.

### Reference comparison (from CMS research)

This mirrors the pattern seen in Wikipedia MediaWiki (substitution on save vs live transclusion), Notion synced blocks (live reference), and WordPress reusable blocks (historically transclusion-based, now "pattern" snapshots by default). Pin-on-add aligns with WordPress's current default because it prevents surprise edits from cascading into published content — the right default for safety-critical SOPs.

Sources:
- [Transclusion — Wikipedia](https://en.wikipedia.org/wiki/Transclusion)
- [Content Versioning in Strapi](https://strapi.io/blog/content-versioning-strapi-setup)
- [strapi-plugin-content-versioning (GitHub)](https://github.com/notum-cz/strapi-plugin-content-versioning)

## Worker Walkthrough Impact

**No breakage.** Current code path:

```ts
// SectionContent.tsx — today
if (type.includes('hazard') || type.includes('emergency')) return <HazardContent />
if (type.includes('ppe') || type.includes('protective'))  return <PpeContent />
if (hasSteps) return <StepsContent />
return <DefaultContent />
```

v3.0 adds an outer layer:

```ts
// v3.0: prefer join, fall back to substring
const family = section.section_kind?.render_family ?? inferFromType(section.section_type)
// inferFromType implements today's substring matching verbatim

switch (family) {
  case 'hazard':
  case 'emergency': return <HazardContent />
  case 'ppe':       return <PpeContent />
  case 'steps':     return <StepsContent />
  case 'signoff':   return <SignoffContent />
  case 'content':
  case 'custom':
  default:          return <DefaultContent />
}
```

- Any custom org-specific kind (e.g. "Pre-flight checks") with `render_family = 'content'` renders as DefaultContent with the org's chosen label/icon/colour.
- Any new canonical family we add later (e.g. `signoff`) gets a dedicated renderer in one place.
- Tab labels and ordering come from `section_kinds.label` and `render_priority`, not hard-coded maps. Adding a new kind = DB insert, no React change.

**Offline/Dexie compatibility:** The existing `sections` Dexie table indexes `section_type` as a string. We add `section_kind_id` + cached `section_kind` subobject to the Dexie schema bump (migration version +1). Workers download the section kind alongside the section, so everything resolves offline without extra round-trips.

**PWA sync engine:** No change. `useSopSync` already does bulk fetches of sections by sop_id — adding a left join on `section_kinds` is a single query change.

## TypeScript Typing Strategy

### Recommended: wide `SopSection` type + optional joined `section_kind` + Zod-validated block payloads

```typescript
// src/types/sop.ts
export type SectionRenderFamily =
  | 'hazard'
  | 'ppe'
  | 'steps'
  | 'content'
  | 'signoff'
  | 'emergency'
  | 'custom'

export interface SectionKind {
  id: string
  organisation_id: string | null  // null = global
  slug: string
  label: string
  render_family: SectionRenderFamily
  icon: string | null             // lucide icon name
  color_token: string | null
  render_priority: number
  description: string | null
}

export interface SopSection {
  id: string
  sop_id: string
  section_type: string             // legacy free-text, still populated
  section_kind_id: string | null
  section_kind?: SectionKind       // optional join, populated by useSopDetail
  title: string
  content: string | null
  sort_order: number
  confidence: number | null
  approved: boolean
  created_at: string
  updated_at: string
}

// Block library
export type BlockKindSlug = 'hazard' | 'ppe' | 'step' | 'emergency' | string

export interface Block {
  id: string
  organisation_id: string | null
  kind_slug: BlockKindSlug
  category: string | null
  title: string
  description: string | null
  current_version: number
  archived: boolean
  created_at: string
}

// Per-kind content shape validated with Zod at the application boundary
export interface BlockVersion {
  id: string
  block_id: string
  version: number
  content: BlockContent            // discriminated by kind_slug
  change_note: string | null
  created_at: string
}

export type BlockContent =
  | { kind: 'hazard'; text: string; severity: 'critical' | 'warning' | 'notice' }
  | { kind: 'ppe';    items: string[] }
  | { kind: 'step';   text: string; warning?: string; tip?: string }
  | { kind: 'emergency'; text: string; contacts?: string[] }
  | { kind: 'custom'; data: Record<string, unknown> }  // escape hatch

export interface SopSectionBlock {
  id: string
  section_id: string
  block_id: string
  pinned_version: number
  follow_latest: boolean
  snapshot_content: BlockContent
  overridden: boolean
  update_available: boolean
  sort_order: number
}
```

**Why this shape:**
- **`SopSection` stays wide, not a discriminated union.** Custom sections don't need per-kind type narrowing at the rendering layer — `render_family` drives which component to use. The rendering switch statement narrows, not the type.
- **`BlockContent` IS a discriminated union.** Each block content shape is known at design time (kinds map 1:1 to Zod schemas). This gives type safety in the wizard, parser, and renderer.
- **Snapshot is cached on the junction row**, not the block — so reading an SOP never needs to resolve block versions at query time. Offline-friendly.

### Zod companion schemas

```typescript
// src/lib/validators/blocks.ts
export const HazardBlockContentSchema = z.object({
  kind: z.literal('hazard'),
  text: z.string().min(1),
  severity: z.enum(['critical', 'warning', 'notice']),
})

export const PpeBlockContentSchema = z.object({
  kind: z.literal('ppe'),
  items: z.array(z.string()).min(1),

export const BlockContentSchema = z.discriminatedUnion('kind', [
  HazardBlockContentSchema,
  PpeBlockContentSchema,
  StepBlockContentSchema,
  EmergencyBlockContentSchema,
  CustomBlockContentSchema,
])

```

Validate on every write to `block_versions` (server action) and on every read that crosses the trust boundary (GPT parser output, admin form submission).

## Wizard UX Flow (for context)

1. Admin creates new section → picks kind from dropdown populated by `section_kinds` query (globals + org-specific).
2. Kind selection reveals a "Pick from library" panel that queries `blocks` filtered by `kind_slug` matching the kind's render_family, `organisation_id is null OR organisation_id = currentOrg`, `archived = false`, optionally filtered by `category` matching the SOP's category.
3. Library panel shows up to 10 best matches (full-text search on `search_tokens`), with "Write new" as the first option.
4. Clicking a library item calls `linkBlockToSection(sectionId, blockId)` — server action that reads `blocks.current_version`, fetches that `block_versions.content`, inserts a `sop_section_blocks` row with `pinned_version` + `snapshot_content` populated.
5. Writing new content prompts "Save as reusable block?" checkbox — if checked, creates a `blocks` row + v1 `block_versions` row scoped to the org, then links it.

**This is the extraction pattern** — blocks are "lifted" from a single SOP into the library once an admin realises "I'll use this again". Don't force admins to create library entries upfront.

## Top Pick

**Schema approach:** Option A — keep free-text `section_type`, add `section_kinds` catalog with advisory FK + render metadata, backfill existing rows.

**Blocks data model:** Single `blocks` table + `block_versions` history + `sop_section_blocks` junction. Nullable `organisation_id` for global vs org scoping. Content stored as JSONB, validated with Zod discriminated union per kind slug.

**Versioning:** Pin-on-add default with `snapshot_content` cached on the junction row. Opt-in `follow_latest` for trusted globals. Admin review flow via `update_available` flag for pinned references.

**Why:**
- **Zero breaking change.** Existing `sop_sections.section_type` stays as-is. Existing rendering logic stays as-is (as a fallback). Existing GPT parser output (free-text types) keeps working.
- **Simpler RLS than separate tables.** One table, one policy, proven pattern. No cross-table joins for wizard queries.
- **Offline-safe by default.** `snapshot_content` on the junction row means workers don't resolve block versions at walkthrough time — the section's cached payload contains everything needed.
- **Matches safety-critical defaults.** Pin-on-add means a global block edit can't silently change a published SOP — the admin explicitly reviews changes per SOP. This is the right default for WorkSafe NZ compliance.
- **Incremental TypeScript migration.** `SopSection` stays wide; only `BlockContent` uses discriminated unions where the value is real.

**Risks:**
1. **`section_type` / `section_kind_id` drift.** Free-text type and FK kind can disagree. Mitigation: trigger on insert/update that auto-links when slug matches, and a nightly consistency check query.
2. **Global block editing risk.** A single global block update could flag hundreds of SOPs with `update_available`. Mitigation: batch the propagation job, rate-limit notifications, show a "this affects 300 SOPs" confirmation before save.
3. **JSONB content drift.** Block content shapes are enforced at the Zod layer, not the DB. Old versions may have outdated shapes after schema evolution. Mitigation: version content shape separately (add `content_schema_version` column on `block_versions`) and write forward-compat readers in the app.
4. **Library discoverability.** Admins won't use a library they can't find. The wizard must proactively surface matches, not require a separate "Library" page. Budget UI work accordingly.
5. **GPT parser doesn't know about library.** On upload, GPT still emits free-text sections with no block linkage. Post-parse, a matching step can suggest "3 sections in this SOP match existing blocks — link them?" — useful but not mandatory for v3.0.

## Pitfalls to Avoid

1. **Don't convert `section_type` to an enum.** The current free-text column is already what extensibility needs. Adding an enum would be a regression and force a destructive migration.
2. **Don't store block content on `blocks` directly.** Put it on `block_versions` only. Otherwise "edit" semantics get ambiguous and audit history is impossible.
3. **Don't rely on `organisation_id IS NULL` working with JWT-filtered RLS by accident.** Test the `null OR =` policy explicitly with a read from a second tenant to confirm isolation. A common mistake is writing `organisation_id = :claim` without the NULL branch and wondering why globals are invisible.
4. **Don't skip `snapshot_content` caching.** If you read block content via a join at walkthrough time, offline sync breaks and worker walkthrough latency regresses. The cached snapshot is load-bearing.
5. **Don't let admins edit global blocks from the app.** Write policies that forbid `organisation_id IS NULL` inserts/updates for authenticated roles. Globals are service_role only — seed via migrations and operations scripts.
6. **Don't forget the `overridden` flag.** When an admin edits a block-linked section inline, preserve the `block_id` reference but set `overridden = true`. Otherwise update propagation will silently overwrite the admin's customisation next time the block changes.
7. **Don't propagate block updates synchronously on save.** A global block edit might touch hundreds of SOPs. Use a deferred job (background worker, or at minimum `setTimeout`-style non-blocking admin client call) so the admin's save UX stays fast.
8. **Don't migrate `sop_section_blocks` into `sop_sections` as denormalised fields.** Tempting because it feels simpler, but you lose the per-block `pinned_version` + `follow_latest` semantics and the junction's update tracking.
9. **Don't forget to bump the Dexie schema version.** Workers need to re-cache sections with the new `section_kind_id` field, or they'll render the fallback path forever.
10. **Don't let the wizard block list go unfiltered.** At 500+ blocks, a flat dropdown is useless. Filter by kind_slug + SOP category + full-text search from day one.
11. **Don't forget `search_tokens` generated column index.** Without GIN on the tsvector, library search on large orgs will be slow.
12. **Don't ship without a "who's using this block" view.** Admins editing a block need to see the blast radius before saving.
