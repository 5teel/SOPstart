# v3.0 Research — Layout Editor

## Executive Summary

**Top pick: Puck (`@measured/puck`)** — a React-first, MIT-licensed visual block editor that serializes cleanly to JSON, runs inside Next.js App Router via a catch-all route recipe, and supports multi-viewport previews (360/768/1280) out of the box. It is philosophically aligned with SafeStart's actual constraint: **workers must still walk through SOPs on 5-6" phones, so "freeform canvas" is the wrong paradigm**. A block/component model reflows naturally from desktop to mobile the same way HTML does — this is how Notion, Gutenberg, Webflow, and Builder.io all solve the same problem.

**Critical negative finding — tldraw is disqualified.** tldraw SDK 4.0 (Sept 2025) moved to a dual license that **requires a paid commercial license to run in production**. The hobby license forbids commercial use, and the commercial tier is widely reported at ~USD $6,000/year per team (official page says "contact sales"). For a multi-tenant SaaS targeting NZ SMB manufacturers, this is a non-starter unless explicitly budgeted. A free 100-day trial key is available for evaluation only.

**Secondary recommendation: Craft.js** as a fallback if Puck's opinionated block model proves too constraining. Craft.js is MIT, 8.6k stars, more framework-like (less opinionated), and powers Destack and several other builders — but last significant release was Feb 2025 and maintenance cadence is slower than Puck.

**Key risks with Puck:** (1) Media-query breakpoints behave weirdly inside its preview iframe — you design for viewport, not `max-width`. (2) No built-in Dexie/IndexedDB offline persistence — we must wire it ourselves via `onChange` callback into our existing Dexie + Zustand layer. (3) The "freeform x/y positioning" spec in the ticket is probably the wrong feature to build; this research recommends pushing back on that requirement in favor of a structured-block model with element ordering + alignment, not absolute pixel coordinates.

## Candidates

### Puck (`@measured/puck`, now `puckeditor/puck`)

- **License:** MIT
- **Bundle size (gzipped):** ~60-90 KB for `@measured/puck` core (lazy-loadable; renderer-only path is smaller — you don't ship the editor to workers)
- **Storage format:** JSON document with `{ content: [...blocks], root: {...}, zones: {...} }`. Passed in/out via `data` and `onChange`/`onPublish` callbacks. Host owns persistence — no vendor lock-in.
- **SSR/App Router compat:** First-class. Official Next.js recipe with `app/[...puckPath]/page.tsx` catch-all. Editor component is `'use client'`; renderer can be RSC.
- **Mobile reflow story:** Built-in viewport switcher (default 360/768/1280). Components define their own responsive CSS (Tailwind breakpoints work). Preview renders in same-origin iframe, so **CSS media queries fire against the iframe width, not window width** — design-time preview is accurate. One gotcha: `max-width` media queries don't always behave as expected inside the iframe; prefer container queries or explicit viewport props on components.
- **Offline support:** Not built in, but trivial to wire. Puck is controlled via `data` prop and `onChange` callback — pipe `onChange` through a debounced Dexie write, hydrate `data` from Dexie on mount. Works with existing SafeStart offline infrastructure.
- **Fit for SafeStart:** Excellent. Puck treats the editor as **"your React components, drag-and-droppable"** — which means hazard cards, step cards, photo blocks, warning callouts, and heading bars can all be authored as normal Tailwind+Lucide React components with Zod-validated props, registered into a `config.components` map. The render output at worker-walkthrough time is just `<Render config={...} data={...} />` — same components, no separate "editor view vs production view" drift. This is the single most important architectural win: one component tree, two contexts (author + walkthrough).
- **Known pitfalls:**
  - `max-width` media queries inside the preview iframe are flaky — use viewport-prop-driven conditional rendering for hard breakpoints.
  - No built-in undo/redo at the field level (only coarse-grained) — acceptable for v3.0.
  - Iframe-based preview adds ~50ms mount cost; not an issue at authoring time.
- **Verdict:** **Strong candidate — recommended top pick.**

### Craft.js (`@craftjs/core`)

- **License:** MIT
- **Bundle size (gzipped):** ~50 KB core, but you bring your own UI layer — real-world bundles land ~100-150 KB including toolbox, settings panels, etc.
- **Storage format:** JSON serialization via `query.serialize()` / `Editor.deserialize()`. Each node has `{ type, props, nodes, linkedNodes }`. Format is stable across minor versions.
- **SSR/App Router compat:** Requires `'use client'`. No SSR of the editor itself (uses refs and DOM measurement heavily). Renderer output can be SSR-rendered if you bypass the editor runtime.
- **Mobile reflow story:** Nothing built in. You're expected to style responsively yourself (Craft.js is framework-agnostic about layout). No viewport switcher.
- **Offline support:** Same pattern as Puck — serialize on change, store in Dexie, deserialize on mount.
- **Fit for SafeStart:** Good but more DIY. Craft.js gives you a node tree and drag/drop primitives; you build the toolbox, settings panel, and rendering yourself. More flexible than Puck (can support nested freeform containers, custom drop zones, linked nodes) but 2-3x the implementation effort. Worth it if Puck's block model proves too rigid — not worth it otherwise.
- **Known pitfalls:**
  - Maintenance cadence has slowed — last significant release Feb 2025, 8.6k stars, a few community forks. Project is healthy but not racing.
  - No viewport preview — you need to build one.
  - Documentation is solid but "build your own editor" means weeks of scaffolding vs Puck's days.
- **Verdict:** **Consider — best fallback if Puck blocks.**

### tldraw (SDK 4.0)

- **License:** **Dual license — paid for commercial production use.** Hobby license (free) is non-commercial only and requires a "Made with tldraw" watermark. Commercial license: "contact sales"; community reporting points at ~USD $6,000/year per team with startup pricing potentially available. Production deployments without a valid license key **will not run** (enforced in code as of 4.0).
- **Bundle size (gzipped):** ~300-400 KB with built-in shapes. Heavy.
- **Storage format:** Snapshot JSON (two parts: `document` for shapes/pages/bindings, `session` for camera/UI state). Excellent migration support via `ShapeUtil.migrations`. Probably the cleanest persistence story of any candidate.
- **SSR/App Router compat:** Client-only via dynamic import with `ssr: false`.
- **Mobile reflow story:** None. tldraw is infinite-canvas by design — shapes have absolute coordinates. "Force mobile layout" only changes the editor chrome, not the content layout. A desktop-authored tldraw scene rendered on a 375px phone will require pan/zoom to read. **This is fundamentally wrong for workers walking through SOPs on the factory floor.**
- **Offline support:** First-class. `persistenceKey` prop stores snapshots to IndexedDB automatically; multiplayer is optional.
- **Fit for SafeStart:** **Disqualified on two independent grounds:** (1) licensing cost for a commercial SaaS, (2) infinite-canvas paradigm is the wrong model for walk-through-on-phone content. The library is excellent at what it does — whiteboards, diagrams, design surfaces — but a phone-friendly SOP is not a whiteboard.
- **Known pitfalls:** Licensing trap (will silently work in dev then fail in prod without a key). Large bundle. React 18/19 only.
- **Verdict:** **Skip.** Only revisit if (a) the product spec changes to "free-form diagram annotator for supervisors on desktop only" and (b) the budget has room for $6k/yr.

### Excalidraw (`@excalidraw/excalidraw`)

- **License:** MIT
- **Bundle size (gzipped):** ~400-500 KB (largest of all candidates)
- **Storage format:** JSON scene format — array of elements with absolute coords, plus appState. Well-documented (`loadFromBlob`, `exportToSvg`, etc.).
- **SSR/App Router compat:** No SSR — must dynamic-import with `ssr: false`. Official Excalidraw Next.js guide exists.
- **Mobile reflow story:** None — same infinite-canvas problem as tldraw. Hand-drawn aesthetic also doesn't match SafeStart's industrial PWA design language.
- **Offline support:** Controlled component — persist JSON yourself.
- **Fit for SafeStart:** Poor. Aesthetic mismatch (cartoon/sketch vs industrial), no reflow story, large bundle.
- **Verdict:** **Skip.** Appropriate for brainstorming/whiteboarding products, not SOP authoring.

### Konva + react-konva

- **License:** MIT
- **Bundle size (gzipped):** ~120-150 KB for react-konva + konva-minimal. Can be trimmed.
- **Storage format:** Konva JSON scene graph (`stage.toJSON()`) — absolute coordinates. Works but verbose.
- **SSR/App Router compat:** Requires dynamic import with `ssr: false`; may need `next.config.js` webpack externals for `canvas` package. Well-documented.
- **Mobile reflow story:** None — it's a canvas abstraction, not a layout system. Same infinite-canvas problem.
- **Offline support:** DIY — serialize the stage, store in Dexie.
- **Fit for SafeStart:** Wrong abstraction level. Konva is for building canvas applications (Figma-like tools, image annotators, games). For a page/slide editor where content eventually needs to render as HTML on a phone, you'd be fighting the tool. Also: canvas content isn't selectable, searchable, or accessible — SafeStart's workers need to be able to tap text to confirm, and supervisors may need to screen-read SOPs.
- **Verdict:** **Skip.** Rule out anything canvas-bitmap-based for this use case.

### Fabric.js

- **License:** MIT
- **Bundle size (gzipped):** ~180 KB
- **Storage format:** JSON scene, absolute coords. SVG export supported (nice for print/PDF, irrelevant here).
- **SSR/App Router compat:** Same client-only story as Konva. No official React bindings — community wrappers only.
- **Mobile reflow story:** None.
- **Offline support:** DIY.
- **Fit for SafeStart:** Weaker than Konva. No official React integration, more manual memory management, same accessibility problems as any canvas solution.
- **Verdict:** **Skip.**

### react-grid-layout

- **License:** MIT
- **Bundle size (gzipped):** ~35 KB
- **Storage format:** Array of `{ i, x, y, w, h }` per item, per breakpoint. Trivial to store in JSONB.
- **SSR/App Router compat:** Client component, but lightweight. Works fine.
- **Mobile reflow story:** **Best-in-class for responsive grids.** Native support for per-breakpoint layouts (`lg/md/sm/xs/xxs` defaults at 1200/996/768/480/0 px). You can author desktop, auto-derive mobile, and reviewers get a real responsive result.
- **Offline support:** Trivial — you're storing a tiny JSON array per SOP page.
- **Fit for SafeStart:** Good for structured grid placement but **too rigid** — everything snaps to grid cells, no free-floating callouts, no overlapping elements. Works as a layer inside a larger solution but not as the whole editor.
- **Verdict:** **Consider as a sub-component** inside Puck or Craft.js when an admin wants tighter layout control on a single block. Not the primary editor.

### dnd-kit

- **License:** MIT
- **Bundle size (gzipped):** ~20 KB core
- **Storage format:** You define it — dnd-kit is a primitive.
- **SSR/App Router compat:** Full.
- **Mobile reflow story:** Whatever you build.
- **Offline support:** Whatever you build.
- **Fit for SafeStart:** dnd-kit is a **toolkit, not an editor**. Use it as the underlying drag-drop layer *inside* Puck/Craft.js (both already use it or similar). Shouldn't be evaluated as a standalone option against Puck — different category.
- **Verdict:** **Use indirectly.** Already bundled by higher-level editors. Reach for it directly only for custom micro-interactions (e.g. reordering steps within a card).

### GrapesJS

- **License:** BSD-3
- **Bundle size (gzipped):** ~250 KB+
- **Storage format:** HTML + CSS + components JSON. Markup-native.
- **SSR/App Router compat:** Official `@grapesjs/react` wrapper with Next.js 15+ support.
- **Mobile reflow story:** Built-in Device Manager with desktop/tablet/mobile preview.
- **Offline support:** DIY.
- **Fit for SafeStart:** Powerful but oversized. GrapesJS is targeted at email/newsletter/landing-page CMS use cases — it's a full web builder with DOM-level editing. Puck gives us ~80% of the same value at ~25% of the complexity and bundle size, with a more React-native component model. GrapesJS would require custom Tailwind integration and careful theme isolation to match SafeStart's dark PWA look.
- **Verdict:** **Skip unless** we end up wanting full HTML fidelity for export (PDF/print, email). Not warranted for v3.0.

### Custom SVG + dnd-kit

- **Verdict:** **Skip.** For a 6-week milestone, building a layout editor from scratch is a time sink with no upside over Puck/Craft.js. Only justifiable if we need something truly bespoke that no library supports — which is not the case here.

## Storage Format Recommendation

**Standardize on Puck's JSON document format, stored as `layout_data JSONB` on `sop_sections`.**

```sql
alter table sop_sections
  add column layout_data jsonb,
  add column layout_version int not null default 1;
```

Shape (TypeScript):

```ts
type SopSectionLayout = {
  version: 1;                          // migration pin
  puck: PuckData;                      // Puck's { content, root, zones }
  mobile_override?: PuckData;          // optional per-breakpoint override
  rendered_at: string;                 // ISO — cache-busting hint for workers
};
```

**Why JSONB on `sop_sections` (not a separate `layout_elements` table):**

1. **Atomic authoring.** Saving a section is one row write. No multi-row transactions, no orphaned elements on crash.
2. **Offline-friendly.** Dexie stores one row per draft section. Sync conflict resolution is "last-write-wins on the section" which matches how non-collaborative SOP authoring actually works.
3. **Schema evolution.** `layout_version` pin lets us run forward migrations in code without ALTER TABLE. Puck ships stable JSON, and we can wrap its data in a thin adapter that normalizes on read.
4. **Supabase RLS already works.** Same `org_id` policies as the rest of `sop_sections`. No new policies needed.

**Migration strategy:** Always append new fields, never rename. Keep a `migrations/` folder of pure functions `(v: N) => (v: N+1)` that run in the Puck `onChange` boundary. This is how tldraw does it and it's the right pattern.

## Mobile Reflow Strategy

**Recommendation: block-based authoring with container-query responsive primitives. Reject freeform x/y positioning.**

This is the most important finding in this document. The ticket asks for "free positioning (x/y/width/height)" — **don't build that**. Here's why:

**How every serious desktop-to-mobile tool solves it:**
- **Notion, Confluence, Gutenberg, Linear docs:** Block-based. A "block" is a logical unit (paragraph, image, callout, toggle) that reflows via CSS. No absolute coordinates. You cannot overlap blocks.
- **Webflow:** Block-based with flexbox/grid + per-breakpoint overrides. "Position: absolute" exists but is explicitly discouraged for responsive layouts.
- **Builder.io:** Same — responsive-first component system with breakpoint overrides.
- **Framer (the one exception):** Allows free positioning but then forces you to manually author separate mobile layouts, and even Framer sites routinely break at unusual aspect ratios.

**Why freeform canvas fails for SOPs:**
1. A "hazard warning at x=420, y=180, width=300px" on desktop becomes unreadable overlap on a 375px phone. You can't rescale proportionally because text doesn't scale that way.
2. Glove-friendly tap targets demand minimum sizes that don't fit arbitrary layouts.
3. Workers walking SOPs step-by-step need linear reading order — that's the whole point of a walkthrough mode.
4. Offline PWA bundle size goes up dramatically if we ship a canvas rendering layer just for workers to read a checklist.

**The right model for SafeStart:**

```
Page = vertical stack of blocks
Block types: heading-bar | text | photo | callout (warning/caution/tip) | step-card | hazard-card | diagram-embed
Block layout: flex column, full width on mobile, optional 2-column row on desktop
Per-block: optional desktop-only width override (50% / 66% / 100%)
```

- **Desktop authors** get rich visual feedback (drag to reorder, drop photos, configure callout severity via form fields) with a 1280px preview showing multi-column layouts.
- **Mobile workers** get the same blocks rendered as a single vertical flow via Tailwind's `md:` breakpoints. No reflow logic needed — CSS does it.
- **Tablet supervisors** fall naturally between the two via `sm:` breakpoints.

This is exactly what Puck's viewport switcher + component-defined responsive props gives us for free. **The "freeform canvas" request in the spec should be gently pushed back on in the roadmap discussion.** If there's a genuine need for a freeform-positioned element (e.g. "annotate this machine diagram with hotspots"), solve that as a single special block type (an image with click-hotspots) — don't make the whole editor freeform.

## Integration with sop_sections

**Recommended: Option A (Hybrid) — one section = one Puck document, but Puck blocks wrap existing SopStep/SopImage entities by ID.**

```
Section row (sop_sections)
├─ layout_data (JSONB) = Puck document
│   └─ content: [
│        { type: "HeadingBlock",  props: { text: "Lockout Procedure" } },
│        { type: "StepBlock",     props: { stepId: "uuid-step-1" } },      ← references sop_steps
│        { type: "PhotoBlock",    props: { imageId: "uuid-img-1" } },      ← references sop_images
│        { type: "HazardBlock",   props: { severity: "danger", text: "..." } },
│        { type: "StepBlock",     props: { stepId: "uuid-step-2" } },
│      ]
└─ sop_steps (FK → sections) unchanged — still queryable, still has photo_required, still feeds assignments
```

**Why hybrid beats the alternatives:**

- **Option 1 (layout = JSONB on section only):** You'd lose the ability to query "which SOPs contain step X" or "which steps require photos across the org." Breaks existing assignment/completion model.
- **Option 2 (layout elements as first-class rows):** You'd duplicate 80% of Puck's state machine in Postgres and lose the ability to save drafts atomically. Rejected.
- **Option 3 (hybrid):** Layout is the *presentation* layer; `sop_steps`/`sop_images` remain the *content* layer. Walkthrough mode can ignore layout entirely and fall back to linear step rendering. Analytics, search, and assignments keep working. Migration from existing SOPs is trivial — the default layout is just `content: steps.map(s => ({ type: "StepBlock", props: { stepId: s.id } }))`.

**Walkthrough fallback:** Worker mode can choose between two renderers:
- `<PuckRender data={section.layout_data} config={...} />` — rich layout if designed
- `<LinearStepList steps={section.steps} />` — fallback if no layout or if device is very narrow

Default is Puck render; fallback is opt-in or automatic under 320px.

## Performance at 50-500 SOPs per org scale

Puck instantiates one editor at a time (per browser tab), and you only load the editor bundle on the admin authoring route — workers never see it. Key performance notes:

- **Editor bundle (admin only):** ~90 KB gzipped + your component code. Lazy-load via `next/dynamic` on the admin route to keep worker bundles clean.
- **Renderer bundle (worker-facing):** Puck's `<Render>` is tree-shakeable and much smaller. Ship only the components actually referenced in layouts — dynamic import per block type if needed.
- **Per-org scale:** 500 SOPs × 10 sections avg = 5,000 section rows with JSONB. Postgres handles this trivially. Individual layout JSON blobs should be capped at ~500 KB each (Puck docs are usually 5-50 KB in practice).
- **Draft memory:** Zustand + Dexie store only the currently-open draft, not all 500. No memory issues.
- **Photo handling:** Continue using Supabase Storage + `sop_images` refs. Puck blocks reference image IDs, never inline base64.

**Flag to monitor:** If the admin dashboard ever loads all 500 layouts at once (e.g. a "search across all SOPs" feature), we need pagination or server-side search — not a client load. This is a normal Postgres/RLS concern, not a Puck concern.

## Offline authoring

Puck is a controlled component: `<Puck data={...} onChange={...} />`. Wire it up as:

```ts
// admin/sops/[sopId]/edit/page.tsx
'use client';
const [data, setData] = useState(initialDataFromDexie);

const debouncedSave = useDebouncedCallback(async (next: PuckData) => {
  await dexie.draftLayouts.put({ sopSectionId, data: next, updatedAt: Date.now() });
  // Background sync to Supabase when online
  if (navigator.onLine) {
    queueSectionSync(sopSectionId);
  }
}, 400);

return <Puck config={config} data={data} onChange={(next) => {
  setData(next);
  debouncedSave(next);
}} />;
```

- **Dexie table:** `draftLayouts` keyed by `sopSectionId`. Reuse the existing Dexie instance from `src/lib/offline/db.ts`.
- **Sync engine:** Extend the existing `sync-engine.ts` pattern — draft layouts sync the same way photo queue does.
- **Conflict resolution:** Last-write-wins per section is fine for v3.0 (single admin editing a draft at a time). If two admins ever edit the same SOP section offline, surface a conflict UI in the sync engine; don't try to CRDT-merge layout JSON.
- **PWA caching:** Admin editor route should be `NetworkFirst` (not offline-first) — admins generally have desk connectivity. Worker walkthrough stays `CacheFirst` for the rendered output.

**Note:** tldraw and react-konva both have simpler auto-persistence (just give them a `persistenceKey`), but we're already building on Dexie/Zustand and the wire-up for Puck is ~15 lines. Not a differentiator.

## Top Pick

**Library:** Puck (`@measured/puck` / `puckeditor/puck`)

**Why:**
1. **Philosophically correct** for SafeStart's actual use case: structured blocks that reflow on mobile, not an infinite canvas that doesn't.
2. **One component tree, two contexts.** The same React components (HazardCard, StepBlock, PhotoBlock) render in both the admin editor and the worker walkthrough. No drift, no double maintenance.
3. **MIT licensed, no vendor lock-in.** Plain JSON out, plain JSON in.
4. **Next.js App Router first-class.** Official recipe, no SSR hacks needed (editor is client, renderer can be server).
5. **Active development.** 11.7k stars, regular releases, responsive maintainer, strong Next.js/Tailwind community content as of 2026.
6. **Integrates cleanly** with existing TanStack Query, Zustand, Dexie, React Hook Form + Zod, Tailwind stack — Puck doesn't impose its own state or styling systems.

**Risks:**
1. **Freeform positioning not supported.** Spec mismatch — needs to be resolved in the roadmap conversation. Recommendation is to drop that requirement; if it survives, see fallback below.
2. **Iframe `max-width` media query quirks.** Use container queries or Puck's viewport-prop overrides instead.
3. **No built-in offline persistence.** ~15 lines of Dexie glue (see Offline section).
4. **Block model requires component pre-registration.** Admins cannot invent new block types at runtime — they get whatever we register in `config.components`. This is actually a *feature* for SafeStart (controlled vocabulary = predictable safety SOPs) but worth noting.

**Fallback: Craft.js** — if (a) we must support true freeform positioning and (b) stakeholders reject the block-based argument above. Craft.js gives you nested draggable nodes with custom containers, and you can build a "freeform container" component that holds absolutely-positioned children. Cost: ~2-3x implementation time, slower release cadence, more DIY scaffolding, still the same mobile reflow problem to solve manually.

**Do NOT use:** tldraw (license), Excalidraw (aesthetic + reflow), Konva/Fabric (wrong abstraction, accessibility), GrapesJS (oversized).

## Pitfalls to Avoid

1. **Don't build freeform x/y positioning.** It doesn't reflow to mobile. Workers walking SOPs on a 5" phone will hate it. Every successful desktop→mobile authoring tool uses blocks, not canvas coordinates. Push back on the spec if needed.

2. **Don't ship tldraw without a license key.** SDK 4.0 silently works in dev and fails in production. Even if you disagree with the rest of this research and want tldraw anyway, budget $6k/year and issue the license key before shipping. Do not build a prototype with tldraw assuming "we'll figure out the license later" — the trial is 100 days.

3. **Don't use canvas libraries (Konva, Fabric, Excalidraw) for content that needs to be accessible.** Canvas output is not text-selectable, not screen-reader-readable, not searchable, not indexable. SafeStart SOPs contain safety-critical text that workers must be able to tap, zoom, and screen-read. Canvas is the wrong layer.

4. **Don't store layout as a separate `layout_elements` table.** You'll lose atomic saves, fight drift between layout and steps, and duplicate Puck's state machine in Postgres. Use JSONB on `sop_sections` with a `layout_version` pin.

5. **Don't skip the worker-mode fallback renderer.** If a section has no layout (legacy SOP, migration not run, mobile device too narrow), workers still need to walk it. Always have a `<LinearStepList>` renderer as a fallback path. Default: Puck render if `layout_data` present and viewport ≥ 320px, else linear.

6. **Don't embed images as base64 in Puck props.** Reference `sop_images.id` instead. Keeps Puck documents small, preserves existing Supabase Storage workflow, lets photo queue/compression still run, and RLS policies continue to gate access.

7. **Don't let admins author blocks inline without Zod validation.** Puck fields are schema-driven — wrap every block's `fields` definition in a Zod schema mirroring the production types. Otherwise you'll get "admin typed nonsense into a number field" crashes at worker walkthrough time.

8. **Don't forget the iframe preview breakpoint quirk.** Puck's preview iframe is same-origin but has its own width. Test breakpoints inside the preview, not by resizing the browser window — they are not the same.
