# v3.0 Research — Collaborative Multi-Admin SOP Editing

## Executive Summary

**Top pick: Hybrid — Pessimistic section-level locking (v3.0) with a clear migration path to CRDT (y-dexie/Yjs) in v3.1+.**

The SafeStart problem space (5-20 admins, 50-500 SOPs, structured step data, rare true simultaneity, heavy offline requirements) does **not** need Google-Docs-grade collaborative editing. The dominant pattern for small teams editing structured content (Payload CMS, Strapi, classic CMS editors) is **field/section-level pessimistic locking with presence indicators**, not CRDTs.

However, one finding changes the calculus: **Dexie (which SafeStart already uses for offline storage) shipped a first-class Yjs addon (`y-dexie`) in late 2024**, which is maintained by the Dexie team themselves and production since November 2024. This makes the "evolve to CRDT later" story dramatically cheaper than it would be on any other stack — because SafeStart already owns the only infrastructure dependency (Dexie/IndexedDB) that would need to change.

**Recommendation path:**
1. **v3.0 ship**: Section-level pessimistic lock on `sop_sections` rows, Supabase Realtime `presence` for "Alice is editing section 3" indicators, optimistic `version` column as a safety net for the lock-release race, and a simple "your draft was saved offline — merge with latest version?" banner for offline reconciliation.
2. **v3.1+ evolve**: If real concurrent-author demand emerges (evidence: admins complaining about lock waits, or customers asking for Google-Docs-like editing), migrate `sop_sections.content` to a `Y.Doc` field via `y-dexie` + `y-postgresql`-style BYTEA storage. Schema additive, no rewrite.

Do **not** ship raw Yjs + y-supabase in v3.0. `y-supabase` is single-maintainer, pre-production, and explicitly not recommended for production by its own author ([y-supabase repo](https://github.com/AlexDunmow/y-supabase)).

## Detailed Evaluation

### 1. Pessimistic Section-Level Locking (RECOMMENDED)

**The idea:** Lock at the `sop_sections` row, not the whole SOP. When Alice clicks "edit section 3", we stamp `locked_by = alice_user_id` and `locked_until = now() + 5 minutes`. Other admins see section 3 read-only with a "being edited by Alice" banner. Lock auto-releases on save, navigate-away, or lease expiry. Other sections of the same SOP remain freely editable.

**Schema additions:**
```sql
ALTER TABLE sop_sections
  ADD COLUMN locked_by uuid REFERENCES auth.users(id),
  ADD COLUMN locked_at timestamptz,
  ADD COLUMN lock_expires_at timestamptz,
  ADD COLUMN version integer NOT NULL DEFAULT 1;

-- Helpful index for lock lookups
CREATE INDEX idx_sop_sections_locked ON sop_sections(sop_id, locked_by) WHERE locked_by IS NOT NULL;

-- RLS: only lock owner can update a locked row
CREATE POLICY "only lock holder can update section"
  ON sop_sections FOR UPDATE
  USING (
    locked_by IS NULL
    OR locked_by = auth.uid()
    OR lock_expires_at < now()
  );
```

**Server action sketch:**
```ts
// src/actions/section-lock.ts
export async function acquireSectionLock(sectionId: string) {
  const supabase = createServerClient();
  const { data: user } = await supabase.auth.getUser();

  // Atomic lock acquisition: only succeeds if unlocked or expired
  const { data, error } = await supabase
    .from('sop_sections')
    .update({
      locked_by: user.user.id,
      locked_at: new Date().toISOString(),
      lock_expires_at: new Date(Date.now() + 5 * 60_000).toISOString(),
    })
    .eq('id', sectionId)
    .or(`locked_by.is.null,lock_expires_at.lt.${new Date().toISOString()}`)
    .select()
    .single();

  if (error || !data) {
    // Someone else holds the lock — fetch current holder for UX
    const { data: current } = await supabase
      .from('sop_sections')
      .select('locked_by, lock_expires_at, profiles:locked_by(full_name)')
      .eq('id', sectionId)
      .single();
    return { ok: false, heldBy: current?.profiles?.full_name };
  }
  return { ok: true, lock: data };
}

export async function releaseSectionLock(sectionId: string) { /* ... */ }
export async function heartbeatSectionLock(sectionId: string) { /* extends lock_expires_at by 5 min */ }
```

**Pros:**
- Zero new dependencies, zero bundle cost.
- Uses infrastructure SafeStart already has (RLS, Supabase, server actions).
- Offline-friendly: an offline admin can still edit their locally cached section; on reconnect we attempt to acquire the lock — if it fails, we show the "merge with Bob's changes" banner.
- Conflict UX is unambiguous and non-scary for factory-floor admins.
- Presence + lock information is already the whole story; no separate presence channel needed beyond Supabase Realtime for the "live" indicator.
- Sections are the natural editing unit in the SafeStart data model already — each section maps to one editor panel.

**Cons:**
- Alice cannot edit the same section as Bob simultaneously. For 5-20 admins editing 50-500 SOPs, section-level conflicts are rare enough that this is acceptable.
- Requires lease expiry + heartbeat logic to avoid stuck locks when browsers crash. Solved by the `lock_expires_at` pattern above + a heartbeat every 60s.
- Doesn't handle "I edited this step offline for 2 hours" gracefully — needs supplementary optimistic version check.

**Mitigation for the offline-2-hours case:**
Combine the lock with an optimistic `version` column. On reconnect:
1. Client attempts to acquire the lock. If another admin has published changes since, `current_version > local_version`, and we show: "This section was updated by Bob while you were offline. Your changes (preview) — keep yours / discard / manually merge."
2. Even with the lock acquired, if `current_version > cached_version`, we force the diff-merge flow.

This is the cleanest, most honest offline story for a small-team editor. It's also how Payload CMS and several Strapi-based editors handle the same problem ([Payload discussion #1490](https://github.com/payloadcms/payload/discussions/1490)).

### 2. Optimistic Lock + Last-Writer-Wins

**Rejected.** Silent data loss is unacceptable for safety-critical SOPs. Imagine Alice adds a "wear safety goggles" step, Bob saves a typo fix 30 seconds later, and Alice's safety step vanishes. This is a regulatory/liability risk for blue-collar NZ factories.

---

### 3. Optimistic Lock + 3-Way Merge (Diff Resolution UI)

Possible, but the UX for non-technical users is poor. GitHub-style diff-merge screens intimidate factory-floor managers. If we're going to invest in sophisticated conflict handling, CRDTs give us automatic merges at similar complexity cost. Rejected as a middle ground that gets the worst of both worlds.

---

### 4. CRDT — Yjs via y-dexie (RECOMMENDED FOR v3.1+)

**The critical finding:** Dexie.js — which SafeStart already uses for offline storage — released a first-class Yjs addon (`y-dexie`) that is maintained by the Dexie team and has been in Dexie Cloud production since November 2024 ([Dexie Global Hackathon 25](https://dexie.org/hackathon/), [y-dexie README](https://github.com/dexie/Dexie.js/blob/master/addons/y-dexie/README.md)).

**Why this matters for SafeStart specifically:** Every other stack has to introduce a new IndexedDB store (via `y-indexeddb`) that lives alongside their existing offline store, creating two databases that need independent quota management, migration schemas, and reconciliation with server state. SafeStart's offline infrastructure is already Dexie — **`y-dexie` turns Yjs into a native Dexie field type**. That eliminates the single biggest architectural objection to CRDTs on a PWA with existing offline code.

**Schema sketch (future state):**
```ts
// src/lib/offline/db.ts
import Dexie from 'dexie';
import { yDexie, Y } from 'y-dexie';

const db = new Dexie('safestart', { addons: [yDexie] });
db.version(5).stores({
  sops: '++id, orgId, status',
  sopSections: '++id, sopId, sortOrder, content: Y.Doc', // <- native Y.Doc column
  photoQueue: '++id, sopId, status',
});
```

**Server persistence:** Yjs binary updates stored as BYTEA in Postgres, broadcast via Supabase Realtime Broadcast channel (not postgres_changes — broadcast is preferred for high-frequency updates to avoid WAL overhead). Pattern is well-established via `y-postgresql` and documented in [PowerSync's Yjs+Postgres guide](https://www.powersync.com/blog/postgres-and-yjs-crdt-collaborative-text-editing-using-powersync).

```sql
CREATE TABLE sop_section_updates (
  id bigserial PRIMARY KEY,
  section_id uuid NOT NULL REFERENCES sop_sections(id) ON DELETE CASCADE,
  update_bytes bytea NOT NULL,
  created_by uuid REFERENCES auth.users(id),
  created_at timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX ON sop_section_updates(section_id, created_at);
-- Periodic compaction: merge N updates into one, delete originals, same pattern as y-postgresql
```

**Pros:**
- True offline-first. An admin edits for 2 hours offline; on reconnect, their local Yjs updates sync, merge automatically with everyone else's, no conflict UI needed.
- No locks, no presence-exclusion UX.
- Local undo/redo works natively — Yjs has `Y.UndoManager` that only undoes the local user's operations without stomping peer edits.
- Same Dexie database — no dual-store, no quota fight, uses the existing migration mechanism.

**Cons:**
- ~40-60 KB gzipped added to the admin bundle (yjs core ~30 KB gzipped historically, plus y-dexie addon and a Supabase Realtime broadcast provider). Verify with Bundlephobia before shipping; size numbers may have grown. [Bundlephobia yjs](https://bundlephobia.com/package/yjs).
- Conflict-free in the CRDT sense does NOT mean "semantically always correct". Example: Alice deletes step 5 while Bob edits step 5 — CRDT merges without conflict, but the merged result may be a zombie edit on a deleted step. Needs UX handling for intentional semantic conflicts.
- The Supabase Realtime Broadcast provider is either custom-built (few hundred LOC, not scary) or via third-party packages like `y-supabase` (NOT production-ready — single maintainer, author says not to use in prod).
- RLS alignment takes care: the "who can write Y updates to which section" check has to live in a server action or edge function that validates before inserting to `sop_section_updates`, because RLS can't introspect binary updates.
- Garbage collection of old updates requires a background compaction job.

**Why defer to v3.1:**
1. Uncertainty about demand. We don't yet know that 5-20 admins actually want real concurrent editing. Section-level locking may fully satisfy the need for 12-18 months.
2. Bundle cost is non-trivial on a PWA. We'd add ~50 KB to every admin page. Worth it if real-time co-editing becomes a headline feature; wasted if it's only used twice a year.
3. Automatic merges have subtle semantic hazards (the delete+edit example above) that need thoughtful UX, and v3.0 is packed with non-collaborative features.
4. The migration path is additive — we can add the `sop_section_updates` table later without touching `sop_sections.content`, and keep the pessimistic lock as a fallback for clients that can't run the CRDT layer.

### 5. CRDT — Automerge 2.x

**Rejected for v3.0.** No Dexie integration — we'd be maintaining two offline stores. Bundle cost ~70-100 KB gzipped (WASM). Automerge's JSON-document model is actually closer to SafeStart's nested SOP→Section→Step structure than Yjs is, so from a data-model purity standpoint Automerge is attractive — but the operational cost of running two offline stores alongside our existing Dexie infrastructure is worse than the data-model win. ([Velt 2025 CRDT library review](https://velt.dev/blog/best-crdt-libraries-real-time-data-sync), [Hacker News discussion on Automerge/Yjs tradeoffs](https://news.ycombinator.com/item?id=29507948))

---

### 6. CRDT — Loro

**Rejected for v3.0.** Loro's own documentation as of early 2026 still advises against production use; API and encoding schema are "experimental" per [Velt's 2025 CRDT survey](https://velt.dev/blog/best-crdt-libraries-real-time-data-sync). Great performance numbers, but not where SafeStart should be taking risk on foundational infra.

---

### 7. Operational Transform (OT)

**Rejected.** OT is what Google Docs uses ([System Design Newsletter on Google Docs](https://newsletter.systemdesign.one/p/how-does-google-docs-work)), but it requires a central server to transform operations in a specific order, which conflicts badly with offline-first. It is also notorious for implementation complexity — Google's own OT engine is one of the harder pieces of software in Google Docs. The industry has broadly moved to CRDTs for new projects.

---

### 8. Custom Supabase Patch Protocol

**Rejected.** This is reinventing either OT or a CRDT, badly, with no library support. Any team who has tried this has eventually rewritten on top of Yjs/Automerge.

## Supabase Integration Pattern

**For v3.0 (pessimistic lock):**

Table schema: additions above on `sop_sections`. No new tables needed.

Realtime channel: one Supabase Realtime channel per SOP for presence — `sop:${sopId}`. Each admin broadcasts `{ userId, sectionId, activity: "editing"|"viewing", at: iso }`. The `sop_section_card` UI subscribes and shows a pill avatar.

```ts
// Presence broadcast for section-level "who is here" indicators
const channel = supabase.channel(`sop:${sopId}`, {
  config: { presence: { key: user.id } },
});
channel
  .on('presence', { event: 'sync' }, () => {
    const state = channel.presenceState();
    // derive per-section occupancy for the UI
  })
  .subscribe(async (status) => {
    if (status === 'SUBSCRIBED') {
      await channel.track({ sectionId: currentSectionId, activity: 'editing' });
    }
  });
```

RLS: The section-lock policy above (only holder can write). Presence data is org-scoped via authenticated-only subscriptions, no RLS on presence itself.

**For v3.1 (Yjs CRDT future state):**

```sql
CREATE TABLE sop_section_yupdates (
  id bigserial PRIMARY KEY,
  section_id uuid NOT NULL REFERENCES sop_sections(id) ON DELETE CASCADE,
  update_bytes bytea NOT NULL,
  created_by uuid REFERENCES auth.users(id),
  created_at timestamptz NOT NULL DEFAULT now()
);

CREATE POLICY "org members can insert updates for their org's sections"
  ON sop_section_yupdates FOR INSERT
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM sop_sections s
      JOIN sops ON sops.id = s.sop_id
      WHERE s.id = section_id
        AND sops.org_id IN (SELECT org_id FROM org_members WHERE user_id = auth.uid())
    )
  );
```

Transport: Supabase Realtime **Broadcast** channel (NOT `postgres_changes`) — broadcast is designed for high-frequency ephemeral messages and avoids WAL overhead. Clients broadcast Yjs updates to each other immediately, and also call a server action that persists the update to `sop_section_yupdates`. Compaction runs periodically via a scheduled edge function. Pattern documented in [Supabase discussion #27105](https://github.com/orgs/supabase/discussions/27105).

## Offline-First Story

### v3.0 (pessimistic lock) offline flow

1. Admin opens a draft SOP while online. Their cached Dexie copy has `version: N` and whatever section lock state exists.
2. Admin goes offline, edits sections 2 and 4. Locally, we mark these rows as `_dirty: true` in Dexie (already how SafeStart handles offline worker completions per `src/lib/offline/sync-engine.ts` patterns).
3. Admin comes back online. Sync engine attempts to acquire locks for sections 2 and 4:
   - **Lock available + version unchanged** → push changes, release lock, done. No UI.
   - **Lock available + version changed** → show "Section 4 was updated by Bob (2 hours ago) while you were offline. [Keep mine] [Keep Bob's] [Show me the difference]" modal. The "Show me the difference" option renders a simple side-by-side of the two versions (not a git-style diff — a human-readable comparison).
   - **Lock held by someone else** → queue the change, show "Waiting for Alice to finish editing section 4… (she started 3 minutes ago)" with a manual retry button.

This is the cleanest offline story we can build without CRDTs. It explicitly accepts that true offline concurrent editing will occasionally need human conflict resolution, which is fine at this team size.

### v3.1 (CRDT) offline flow

With `y-dexie` + Yjs, the offline story becomes trivial: local edits are recorded as Yjs updates in IndexedDB via `y-dexie`. On reconnect, the client loads all remote updates since its last sync, applies them, and broadcasts its own. Merges are automatic. No modal. No "your changes may have been lost" message. This is the ideal endgame.

### Critical non-break guarantee

Both approaches **preserve the existing worker offline story** (SOP walkthrough, photo capture, completion records) because collaborative editing only runs on `admin/` routes. Workers never touch the CRDT or lock-acquisition code. The v3.1 CRDT migration can be scoped to `sop_sections.content` only — every other Dexie store (photo queue, completion store, assigned SOPs) stays exactly as it is.

## Presence Awareness

**v3.0:** Supabase Realtime `presence` feature ([Supabase Presence docs](https://supabase.com/docs/guides/realtime/presence)) handles this natively. Each admin broadcasts `{ sectionId, avatar, name }` on `sop:${sopId}` channel. UI shows avatar pills on section cards. No extra infra.

**v3.1 CRDT:** Yjs has `Awareness` protocol built-in — cursor positions, user colors, selection ranges, all shippable over the same Supabase broadcast channel used for Yjs updates. More granular (shows exact cursor position) but also more to build.

**Recommendation:** Start with Supabase presence in v3.0. It's 30 lines of code and gets you 80% of the UX.

---

## Undo / Redo

**v3.0 pessimistic lock:** Local undo/redo works out of the box because you're the only writer while the lock is held. A simple client-side undo stack in Zustand (or `react-hook-form`'s built-in dirty tracking) covers this. No concurrency to worry about.

**v3.1 CRDT:** Yjs ships `Y.UndoManager`, which scopes undo to the local user's operations only — hitting Ctrl+Z never reverts something Alice typed. This is one of the specific reasons CRDT libraries exist; rolling your own on top of OT or patches is notoriously hard.

## Conflict UX Mockups (v3.0)

### Case A: "Section locked" (another admin is editing right now)

```
┌────────────────────────────────────────────────────┐
│ [SECTION 3: Glass Loading Procedure]               │
│                                                    │
│ 🟡 Alice Chen is editing this section              │
│    Started 2 minutes ago                           │
│                                                    │
│ [View read-only]   [Ask Alice to finish]           │
└────────────────────────────────────────────────────┘
```

No diff UI. No forms. Just a clear "someone else is here, wait or look at it read-only" message. The "Ask Alice to finish" button sends a Supabase broadcast notification to Alice.

### Case B: "Your offline changes conflict" (reconnect + version mismatch)

```
┌────────────────────────────────────────────────────┐
│ ⚠️ Section 4 changed while you were offline        │
│                                                    │
│ Bob Smith updated this section 2 hours ago while   │
│ you were working offline. We kept both versions:   │
│                                                    │
│   Your version:        Bob's version (now live):   │
│   ┌──────────────┐     ┌──────────────┐            │

```

Three buttons. Non-technical language. No git-style three-way diff. The "merge manually" path opens Bob's version in the editor with Alice's version pinned in a side panel for reference.

### Case C: "Lock expired while you were offline"

Lock auto-released after 5 minutes, no one else took it, no one else changed the version. Silently re-acquire the lock on reconnect, no UI.

## Top Pick

**Approach:** Section-level pessimistic locking with Supabase Realtime presence, plus an optimistic `version` column as an offline safety net.

**Why:**
1. It is the simplest approach that actually solves scenarios A, B, and C cleanly.
2. It ships with zero new dependencies and ~0 KB bundle cost.
3. It handles the offline-2-hours factory floor case via the `version` column safety net, without needing CRDT infrastructure.
4. Scenario D (same-section conflicts) is acknowledged honestly with the diff modal, not hidden behind "CRDT auto-merged" (which can produce semantically wrong merges anyway).
5. The migration to CRDT is genuinely additive later — `sop_sections.content` becomes a `Y.Doc` field via `y-dexie`, the lock column becomes unused but non-breaking, and the conflict modal disappears.

**Risks:**
- **Stuck locks.** Browser crashes, admin closes the tab without clicking save, Wi-Fi drops mid-edit. **Mitigation:** `lock_expires_at` lease of 5 minutes + client heartbeat every 60 seconds + a "force release" button for SOP admins who have elevated permissions + a nightly cron to clear abandoned locks older than 1 hour.
- **The "lock was released while I typed" race.** Alice's lease expires at minute 5, Bob grabs the lock at minute 5:01, Alice finishes typing and clicks save at minute 5:02. **Mitigation:** `version` column check on save — Alice's save fails with "Bob took over this section", we offer the diff modal.
- **Field-level granularity demand.** Two admins may want to edit different *steps* within the same section simultaneously. Initial implementation locks the whole section, which is fine if sections have 3-7 steps. If users complain, push the lock down to `sop_steps` rows (additive change, no migration).
- **Presence channel scale.** If org has 50 admins all opening the same SOP, presence broadcasts get chatty. Supabase Realtime handles this fine at SafeStart's scale (5-20 admins per org).

**Fallback:** If pessimistic locking proves to frustrate users within 3 months of shipping, we have a clean escalation path:
1. First: push lock granularity from section to step.
2. Second: add CRDT for the `content` field of actively-edited sections via `y-dexie`. Lock column stays as a "soft reservation" (visible to UI, not enforced in DB).
3. Third: remove the lock column entirely once CRDT is the authoritative path.

## Phased Rollout Option

**Yes — there is a clean migration path that avoids schema rewrites.**

### Phase 1 (v3.0): Section locking + presence
- `sop_sections` gets 4 new columns (`locked_by`, `locked_at`, `lock_expires_at`, `version`).
- New server actions: `acquireSectionLock`, `releaseSectionLock`, `heartbeatSectionLock`.
- New Supabase Realtime `sop:${sopId}` presence channel subscribed by admin UI.
- Conflict modal component for the offline reconciliation case.
- Bundle delta: essentially zero.

### Phase 2 (v3.1 or v3.2): CRDT for hot sections
- Install `y-dexie`, `yjs`, and a small Supabase broadcast provider (custom ~300 LOC or an audited fork — do NOT depend on `y-supabase` without forking it first).
- Add `sop_section_yupdates` table (BYTEA updates, RLS scoped to org membership).
- Extend Dexie schema with `Y.Doc` column for section content. Migration copies existing `content` text into the Y.Doc.
- Update lock acquisition: if CRDT is active for the section, lock becomes a "soft presence hint" not a hard writer gate.
- Compaction edge function runs daily.
- Bundle delta: ~40-60 KB gzipped on admin routes only (code-split so worker routes are unaffected).

### Phase 3 (v3.3+): Remove the lock column
- After 2-3 months of dogfooding Phase 2, remove the `locked_*` columns.
- Delete the pessimistic lock server actions.

This is additive at every step. No destructive migrations. No data loss risk. The only semi-breaking change is Phase 2's Dexie schema bump, which is handled by Dexie's version migration mechanism and affects admin clients only.

## Pitfalls to Avoid

1. **Do NOT ship `y-supabase` in its current form.** Single-maintainer, pre-production, author explicitly recommends against production use ([y-supabase repo](https://github.com/AlexDunmow/y-supabase)). If Yjs+Supabase is ever needed, fork `y-supabase` into the repo or build a minimal custom broadcast provider (~300 LOC).

2. **Do NOT use `postgres_changes` as the transport for high-frequency collaborative updates.** It multiplies WAL overhead and will degrade the primary database under load. Use Supabase Realtime **Broadcast** for ephemeral collaborative traffic; reserve `postgres_changes` for coarse "SOP published" / "SOP archived" notifications.

3. **Do NOT lock at the whole-SOP level.** It caps throughput at 1 concurrent editor per SOP and makes the product feel single-player. Section-level is the right granularity for SafeStart's data model.

4. **Do NOT assume CRDTs = no conflicts.** They eliminate *merge* conflicts. They do not eliminate *semantic* conflicts (Alice deletes a step while Bob edits it, "edit" ends up attached to a deleted step). CRDT UIs still need tombstone handling. See [Cinapse's "Why we moved away from CRDTs" post-mortem](https://www.powersync.com/blog/why-cinapse-moved-away-from-crdts-for-sync) for a real-world example.

5. **Lock lease expiry must be short.** 5 minutes is the maximum — longer locks cause "stuck" SOPs when browsers crash. Pair with a 60-second heartbeat.

6. **Lock RLS policy must permit the holder to update, not just read.** Easy to get this backwards. Test: can Alice update a section she has locked? Can Bob update a section Alice has locked? Can an unlocked section be updated by anyone with general permission?

7. **Test the offline-reconnect path on slow 3G.** The race window for "my lock just expired, Bob took over, my save hits the server" grows on bad connections. The `version` column safety net catches this, but only if we test it.

8. **Presence channel must not leak cross-org data.** Scope Realtime subscriptions to authenticated, org-scoped channels. An admin from Org A must never see "Alice from Org B is editing section 3" presence.

9. **Undo after lock release is dangerous.** If Alice releases the lock, Bob edits, then Alice reopens and hits Ctrl+Z from her local stack, she would stomp Bob's changes. **Mitigation:** clear the undo history on lock release. When Alice reacquires the lock, she starts with a fresh undo stack scoped to the current state.

10. **Don't let the conflict modal become a wall of code.** Non-technical admins will close the tab and never come back. Three plain-English buttons, visible headline, human-readable comparison.

11. **Photo/image edits complicate all approaches.** SOP sections can contain images. Image changes need to be treated as atomic (either you uploaded the new photo or you didn't) — don't try to CRDT-merge image blob references. Conflict resolution for photo edits should always be "keep newest wins" by default.

12. **Do not break the worker offline path.** All collaborative editing code must live under `admin/` routes only. Workers never run Yjs, never participate in presence channels, never interact with section locks. Test this explicitly by loading the worker walkthrough bundle and confirming no Yjs/lock code is present.

## Confidence & Open Questions

**Confidence: MEDIUM-HIGH.**

- HIGH confidence on: the pessimistic-lock + presence recommendation for v3.0 (well-established pattern across Payload CMS, Strapi, classic CMS editors; Supabase primitives are mature).
- HIGH confidence on: Yjs being the right CRDT library choice *if* we choose CRDTs — the `y-dexie` native integration is a decisive advantage.
- MEDIUM confidence on: exact Yjs bundle numbers for 2025/2026 — Bundlephobia was unreachable during research. Needs hands-on verification before Phase 2 planning.
- MEDIUM confidence on: whether 5-20 admins actually want real-time co-editing. This is a product-research question, not a technical one. Ship v3.0 pessimistic locking and instrument it — measure how often lock contention happens in real usage.

**Open questions for product:**
- Do admins ever need to simultaneously co-edit the same section, or is section-granularity enough?
- Does the conflict modal need to show 3 versions (mine, theirs, common ancestor) or just 2 (mine, theirs)? 2 is much simpler; 3 adds git-like clarity.
- Should SOP admins have a "break lock" elevated permission (for the case where someone else closed their tab 2 minutes ago and we don't want to wait 3 more)?
