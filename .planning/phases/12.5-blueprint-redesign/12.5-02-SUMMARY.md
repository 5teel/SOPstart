## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Security] isSopTabId() guard on URL ?tab= param**
- **Found during:** Task 3
- **Issue:** T-12.5-02-01 in the plan's threat model requires that arbitrary strings in `?tab=` don't reach dispatch logic unchecked.
- **Fix:** `isSopTabId()` narrows the raw search param to the 6-tuple before any routing decision. Falls through to `'overview'` for invalid values. Exported for reuse by page.tsx and any future tab consumers.
- **Files modified:** src/components/sop/SopTabNav.tsx (already in plan spec — confirming it was implemented)

**2. [Rule 1 - Adjustment] EvidenceButton onClick type cast**
- **Found during:** Task 1
- **Issue:** The component props declared `onClick?: MouseEventHandler<HTMLElement>` but when rendering a `<button>`, React expects `MouseEventHandler<HTMLButtonElement>`. TypeScript flagged the type mismatch.
- **Fix:** Cast `onClick` to `MouseEventHandler<HTMLButtonElement>` in the button branch. The `<a>` branch is handled separately with an `href` guard, so the cast is safe.
- **Files modified:** src/components/ui/EvidenceButton.tsx

**3. [Rule 2 - Adjustment] Suspense boundary for useSearchParams**
- **Found during:** Task 3
- **Issue:** The plan noted "wrap the whole SopDetailPage body in Suspense at the export level if needed" — this is required in Next.js App Router. Without it, `useSearchParams()` throws during SSR.
- **Fix:** Split into `SopDetailInner` (the real component using `useActiveTab()`) and `SopDetailPage` (default export wrapping `SopDetailInner` in `<Suspense>`). Pattern matches the plan's own suggested split.
- **Files modified:** src/app/(protected)/sops/[sopId]/page.tsx

## WorkerPreviewClamp placement note

`WorkerPreviewClamp` is placed **at the shell level** in `page.tsx` — it wraps the entire tab body switch (`{active === 'overview' && <OverviewTab>} …`). This means:
- All 6 tabs are clamped uniformly without each tab needing to import `WorkerPreviewClamp`.
- The `FlowTab` uses `<BlueprintCanvas fullBleed>` internally but still gets the 430px outer shell clamp when mobile preview is active — this is intentional for mobile UX consistency.
- Waves 3/4/5 can override or remove the clamp on specific tabs if needed by wrapping the individual tab with a negating `maxWidth: '100%'` override.

## Wave 3+ interface notes

- All tab body stubs accept `{ sop: SopWithSections }` — this is the contract Waves 3/4/5 implement against. No refactoring of `page.tsx` call sites needed.
- `SopTabNav` exports `SOP_TABS`, `SopTabId`, `useActiveTab`, `isSopTabId` — Wave 4 walkthrough can use `useActiveTab()` directly to read the current tab from URL.
- `usePreviewStore` is available globally — any new Wave 3+ component needing the viewport state imports from `@/stores/preview`.
