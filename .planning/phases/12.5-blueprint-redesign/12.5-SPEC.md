# Phase 12.5: Blueprint Redesign — Specification

**Created:** 2026-04-24
**Ambiguity score:** 0.13 (gate: ≤ 0.20)
**Requirements:** 11 locked

## Goal

Ship the worker-first engineering-drawing UI: every route under `/sops/[sopId]/*` renders the paper/ink blueprint aesthetic with a unified 6-tab interface (overview/tools/hazards/flow/model/walkthrough), immersive mobile walkthrough, server-side voice capture, command palette, and 7 new AI-accessible block types wired through the three-place contract so `/api/schema` surfaces them automatically.

## Background

**Current worker-facing state (pre-12.5):**
- `src/app/(protected)/sops/[sopId]/page.tsx` renders the SOP detail as a single scrollable list with the existing steel-900 / brand-yellow dark theme.
- `src/app/(protected)/sops/[sopId]/walkthrough/` is a step-by-step list walkthrough — same dark theme, no tabs, no immersive mobile card, no voice input, no command palette.
- `src/components/sop/SectionContent.tsx` dispatches to legacy per-family renderers (`HazardContent`, `PpeContent`, `StepsContent`, `DefaultContent`) or to `LayoutRenderer` when a section has `layout_data`. All rendering uses steel-900 backgrounds.
- `src/app/(auth)/login/page.tsx` + `src/app/page.tsx` are minimal brand-yellow CTAs — also dark theme.
- The Phase 12 admin builder at `/admin/sops/builder/[sopId]` intentionally keeps the dark theme; this phase does NOT touch admin surfaces.

**Current block inventory (shipped in Phase 12):**
- 7 blocks: `TextBlock`, `HeadingBlock`, `PhotoBlock`, `CalloutBlock`, `StepBlock`, `HazardCardBlock`, `PPECardBlock` + `UnsupportedBlockPlaceholder` placeholder.
- All wired through the three-place contract (`puckConfig.components` + `BLOCK_REGISTRY` + `BlockContentSchema`).
- `/api/schema` at `src/app/api/schema/route.ts` returns JSON-Schema for these 7, plus 8 string enums and layout envelope.

**Current voice / cmdk / flow-graph state:**
- No voice capture anywhere in the app.
- No command palette.
- No flow-graph visualisation — sections are only ever rendered linearly.
- No `sops.flow_graph` column.
- No `sop_voice_notes` table.
- No preview toggle on worker-facing routes (Phase 12 ships one on the admin builder only).

**Sketch + skill:**
- `sketches/sop-blueprint/index.html` — 2015-line prototype.
- `.claude/skills/sketch-findings-SOPstart/` — auto-loads via `CLAUDE.md` when building UI; contains design tokens, layout primitives, screen inventory, new-block-type specs (with three-place contract mapping), and interaction patterns (voice state machine, cmdk, mobile immersive walkthrough, preview toggle).

## Requirements

1. **SB-UX-01 Paper/ink palette on worker routes**: The blueprint palette replaces the steel-900/brand-yellow dark theme on every `/sops/*` route and the public landing.
   - Current: `src/app/page.tsx`, `src/app/(protected)/sops/[sopId]/page.tsx`, `src/app/(protected)/sops/[sopId]/walkthrough/` all render the dark theme. No paper/ink CSS tokens exist.
   - Target: All `/sops/*` routes + `/` + `/login` + `/sign-up` use `--paper` (#fafafa) backgrounds, `--ink-900` (#09090b) text, JetBrains Mono + Inter fonts, and the 6 semantic accent colours only for their semantic role. Admin routes under `/admin/*` keep the existing dark theme unchanged.
   - Acceptance: A DOM snapshot of `/sops/[sopId]` returns `background-color: rgb(250, 250, 250)` (paper) and `color: rgb(9, 9, 11)` (ink-900) on the body; the same snapshot on `/admin/sops` still returns the steel-900 background.

2. **SB-UX-02 Unified 6-tab interface**: Worker SOP detail is reorganised into six tabs sharing a single top chrome with breadcrumb + tab nav.
   - Current: `/sops/[sopId]` renders one scrolling column; there are no tabs for overview / tools / hazards / flow / model / walkthrough.
   - Target: `/sops/[sopId]` mounts a `<TabNav>` with six tabs (overview / tools / hazards / flow / model / walkthrough). Tab state lives in the URL (`?tab=flow`). Top chrome renders product name + SOP title + current tab name + current step (when on walkthrough).
   - Acceptance: `/sops/[sopId]?tab=hazards` renders only the Hazards tab body; switching to `?tab=flow` renders only the Flow tab body; browser back/forward traverses tab history correctly.

3. **SB-UX-03 Immersive mobile walkthrough**: On phones (viewport ≤ 430px), the walkthrough tab renders one full-screen step card at a time; on desktop, both immersive and list modes are available.
   - Current: Walkthrough renders the same scrollable list on all viewports.
   - Target: At viewport ≤ 430px, `/sops/[sopId]?tab=walkthrough&step={id}` renders `<ImmersiveStepCard>` exclusively — a full-screen card with header (breadcrumb + `n/total`), body (step content with `bg-grid`), and sticky PREV/NEXT footer. Above 430px, a `<ViewModeToggle>` lets the user choose between immersive and list modes, with preference persisted in `localStorage` (key: `safestart.walkthrough-mode`).
   - Acceptance: Playwright opens `/sops/{id}/walkthrough` at viewport 390×852 and asserts `.immersive-step-card` is present and `.walkthrough-list` is not rendered. At viewport 1280×720, toggling list mode stores `{mode:'list'}` in localStorage and renders `.walkthrough-list`.

4. **SB-UX-04 Seven new AI-accessible block types**: `MeasurementBlock`, `DecisionBlock`, `EscalateBlock`, `SignOffBlock`, `ZoneBlock`, `InspectBlock`, `VoiceNoteBlock` are registered in all three contract locations (`puckConfig.components` + `BLOCK_REGISTRY` in `src/actions/introspection.ts` + `BlockContentSchema` discriminated union in `src/lib/validators/blocks.ts`). Each exports its own Zod schema in `src/components/sop/blocks/`.
   - Current: Only the 7 Phase-12 blocks are registered. `/api/schema` lists 7 blocks.
   - Target: All 7 new blocks registered. `/api/schema` lists 14 user-authorable blocks (15 including `UnsupportedBlockPlaceholder`). Each new block has a Zod schema, a description, and a minimal valid example surfaced in `/api/schema`.
   - Acceptance: `curl https://{host}/api/schema | jq '.blocks | length'` returns `14` (or `15` including placeholder). `curl https://{host}/api/schema | jq '.blocks[] | .id'` includes `MeasurementBlock`, `DecisionBlock`, `EscalateBlock`, `SignOffBlock`, `ZoneBlock`, `InspectBlock`, `VoiceNoteBlock`. Each returns a valid JSON-Schema `props_schema` with at least one required field.

5. **SB-UX-05 EscalateBlock hybrid mode**: `EscalateBlock` carries a per-instance `escalationMode` prop with three values, default `form`.
   - Current: No `EscalateBlock` exists.
   - Target: `EscalateBlockPropsSchema` includes `escalationMode: z.enum(['alert', 'lock', 'form']).default('form')`. Behaviour:
     - `alert` → writes a row to `worker_notifications` + no walkthrough block (async).
     - `lock` → sets a lock record; NEXT button disabled until a supervisor SignOff unlocks it.
     - `form` → opens a structured escalation report form; writes to new `escalation_reports` table (schema defined in discuss-phase); worker may continue after submit.
   - Acceptance: Three Playwright scenarios — one per mode — each triggers an `EscalateBlock` and asserts (a) the expected side-effect (notification / lock record / form submission), and (b) the worker UX state matches the mode. Default behaviour when no `escalationMode` prop is set on the instance matches the `form` case.

6. **SB-UX-06 Server-side voice capture**: Voice capture records audio in-browser, streams to a server endpoint for transcription, persists the transcript to cloud storage, and populates the target field.
   - Current: No voice capture exists anywhere in the app.
   - Target: `MeasurementBlock` and `VoiceNoteBlock` expose a mic control that cycles through idle → listening → transcribing → captured states. The audio blob is POSTed to a new `/api/voice/transcribe` route (server-side Whisper or equivalent), which returns the transcript. Measurements are numeric-parsed and validated against tolerance; notes are saved to a new `sop_voice_notes` table with link to `sop_completions.id`. Offline: the blob is queued in Dexie with `syncState: 'dirty'` and sent on reconnect.
   - Acceptance: Playwright test with a pre-recorded audio blob (stubbed `MediaRecorder`) uploads to `/api/voice/transcribe`, receives a transcript, and asserts the transcript is visible in the UI within 5s. Triggering the same capture while offline (mocked `navigator.onLine = false`) persists the blob to Dexie `voiceNotesQueue` and reconnect triggers auto-flush.

7. **SB-UX-07 Command palette (cmdk)**: Cmd/Ctrl+K opens a modal palette that searches steps + "Ask AI" (SOP-scoped) + tools/hazards for the current SOP.
   - Current: No command palette.
   - Target: Global `Cmd/Ctrl+K` keyboard handler mounted inside the worker layout. Opens a `<CommandPalette>` modal (640px desktop / full-screen mobile) with three result groups: JUMP TO STEP, ASK AI, TOOLS & HAZARDS. "Ask AI" submits queries to a new `/api/sops/[sopId]/ask` endpoint that is hard-scoped to the current SOP (no cross-SOP context).
   - Acceptance: Pressing `Cmd+K` anywhere under `/sops/*` opens the palette. Typing a known step number jumps to that step in the walkthrough. Submitting an Ask-AI query sends a POST to `/api/sops/{sopId}/ask` — the request body must contain `sopId` and the server MUST reject any request where the `sopId` in the body doesn't match the path.

8. **SB-UX-08 Preview toggle on worker routes**: The desktop ↔ mobile preview toggle (currently only on the admin builder) is available on every worker-facing tab, with preference persisted in `localStorage`.
   - Current: Preview toggle exists only inside Puck on the admin builder via `BUILDER_VIEWPORTS`.
   - Target: A new `<WorkerPreviewToggle>` component renders in the top chrome of the worker layout. Clicking MOBILE clamps the tab body to a 430px-wide column (no skeuomorphic phone frame). Preference saved to `localStorage` key `safestart.preview-viewport`.
   - Acceptance: On desktop at viewport 1280×720, clicking MOBILE wraps the tab body in a 430px container and persists `{viewport:'mobile'}` to localStorage. Reloading the page restores the mobile view. Clicking DESKTOP restores full width and persists `{viewport:'desktop'}`.

9. **SB-UX-09 Flow graph from both derivation and explicit JSONB**: The Flow tab renders a flow graph built by deriving nodes+edges from sections/steps/block-types **AND** honouring an optional explicit `sops.flow_graph` JSONB column when present (explicit overrides derived).
   - Current: No Flow tab; no derivation function; no `sops.flow_graph` column.
   - Target: New migration adds `sops.flow_graph jsonb` (nullable). A new `deriveFlowGraph(sop)` function in `src/lib/sop/flow-graph.ts` walks sections → steps → `layout_data.content[].type` and produces `{ nodes, edges }`. Flow tab checks `sops.flow_graph`: if present, render as-is; if null, render derived output. Node types and colours match the sketch (7 distinct node types).
   - Acceptance: For a SOP with `flow_graph IS NULL`, the Flow tab renders a derived SVG with at least one node per section and edges between sequential steps. Setting `sops.flow_graph` to a valid override via Supabase admin client causes the Flow tab to render the override exactly (no derivation). Setting it to an invalid shape falls back to derivation and logs a console warning once per page load.

10. **SB-UX-10 Flow graph edited inside Puck**: The Flow tab's editing UI is inside the Puck editor (admin builder), not a separate standalone tool.
    - Current: No flow editing. Puck only supports component-based block authoring.
    - Target: A new Puck-level component (`FlowGraphField` — a `type: 'custom'` Zod-backed Puck field registered on the SOP-level layout root, not on individual sections) allows admins to edit the flow-graph JSON for the current SOP from inside the admin builder at `/admin/sops/builder/[sopId]`. Edits write to `sops.flow_graph` via a new server action.
    - Acceptance: Admin opens the builder, opens the SOP-level root fields panel, sees a "Flow graph" field, can add/remove/reposition nodes, saves, and the change is reflected in the worker's `/sops/[sopId]?tab=flow` within the 3s autosave window (same as existing layout_data sync).

11. **SB-UX-11 ModelBlock registered but feature-flagged**: `ModelBlock` is present in the three-place contract (palette + `/api/schema` + discriminated union) but the 3D viewer itself is stubbed behind the `NEXT_PUBLIC_MODEL_BLOCK_ENABLED` feature flag (default `false`).
    - Current: No `ModelBlock`; no 3D viewer dependency (three.js / @react-three/fiber not installed).
    - Target: `ModelBlock` registered in `puckConfig.components` / `BLOCK_REGISTRY` / `BlockContentSchema`. Schema: `z.object({ assetUrl: z.string().url(), hotspots: z.array(...).default([]), defaultLayers: z.array(z.string()).default([]) })`. Component renders a placeholder ("3D viewer disabled — enable `NEXT_PUBLIC_MODEL_BLOCK_ENABLED` to load") when the flag is false. No three.js dependency added in 12.5.
    - Acceptance: `/api/schema` includes `ModelBlock` with a valid `assetUrl` property. Dropping a `ModelBlock` into a SOP with the flag unset renders the placeholder, not a 3D viewer. Enabling the flag in a feature-flag test renders the placeholder text "3D viewer — Phase 12.6 pending" (or equivalent token — finalised in discuss-phase).

## Boundaries

**In scope:**
- Worker-facing route refresh: `/`, `/login`, `/sign-up`, `/sops/*` — all adopt paper/ink palette and blueprint aesthetic.
- 6-tab SOP detail interface (overview / tools / hazards / flow / model / walkthrough).
- Immersive mobile walkthrough + list walkthrough on desktop.
- 7 new block types (Measurement, Decision, Escalate, SignOff, Zone, Inspect, VoiceNote) with full three-place contract wiring.
- Server-side voice transcription endpoint + cloud persistence of voice notes.
- Command palette (cmdk) with step jump, Ask AI (SOP-scoped), tools/hazards search.
- Preview toggle extended from admin builder to worker routes with localStorage persistence.
- Flow tab with derived + explicit-override rendering; `sops.flow_graph` column added.
- Flow graph editing inside Puck (admin builder).
- `ModelBlock` three-place contract registration behind a feature flag.

**Out of scope:**
- ModelBlock 3D viewer implementation (three.js / glb / usdz loader, upload UI, hotspot editor) — deferred, feature-flagged placeholder only in 12.5.
- Reusable block library (`sop_section_blocks` CRUD, Org-vs-global split) — Phase 13.
- AI-drafted SOPs (prompt → GPT draft → Claude verifier) — Phase 14.
- NZ template library — Phase 15.
- Konva image annotation / DiagramHotspotBlock — Phase 16.
- Collaborative editing with section locks — Phase 17.
- Admin builder visual refresh — admin UI keeps steel-900 / brand-yellow. This phase only touches worker-facing routes + the flow-graph Puck field on admin.
- Voice transcription model selection (Whisper variant, language support beyond en-NZ/en-AU/en-US) — discuss-phase locks the provider; this SPEC only requires *some* server-side transcription.
- Offline-mode voice transcription (Whisper WASM on-device) — cloud-first only per user decision; Dexie queues the blob and sends on reconnect but does not transcribe locally.
- Bulk-upsert sections endpoint, file metadata index, enum registry table — P1 items from the earlier AI-surface audit; deferred to Phase 14 planning or later.

## Constraints

- **Three-place contract is mandatory for every new block**: `puckConfig.components` + `BLOCK_REGISTRY` in `src/actions/introspection.ts` + `BlockContentSchema` discriminated union. A test MUST fail the build if any new block is missing from one of the three locations. Source pattern: `src/lib/builder/puck-config.tsx:399-418` comment block.
- **Worker-facing CSS must not duplicate or override admin steel-900 tokens**: the `--paper` / `--ink-*` / `--accent-*` tokens are additive. Keep existing `steel-*` / `brand-yellow` tokens untouched — admin routes still use them.
- **No JS viewport branching inside block components**: SB-LAYOUT-03 invariant from Phase 12 continues to hold. `grep -rE "isMobile|useMediaQuery|navigator\.userAgent" src/components/sop/blocks/` must return zero matches after this phase.
- **Mobile walkthrough must work offline**: step state persists to Dexie + syncs on reconnect. Same offline contract as Phase 12 autosave.
- **`/api/voice/transcribe` is authenticated**: org-scoped via Supabase session. No public access.
- **`/api/sops/[sopId]/ask` is hard-scoped**: the server validates that the path `sopId` matches any `sopId` in the body and rejects cross-SOP queries. No cross-tenant leakage.
- **Default Whisper / transcription language**: `en-NZ` per project NZ-manufacturing target market. Fallbacks via `VoiceNoteBlockPropsSchema.language` enum.
- **ModelBlock feature flag**: `NEXT_PUBLIC_MODEL_BLOCK_ENABLED` defaults `false` in all environments shipped in 12.5. The three.js dependency is NOT added to `package.json` in this phase.
- **Flow graph derivation performance**: `deriveFlowGraph(sop)` must return within 50ms for SOPs with ≤ 500 steps (representative upper bound). Memoise via TanStack Query.
- **`sops.flow_graph` migration is additive**: nullable column, no backfill; legacy SOPs render derived graphs.
