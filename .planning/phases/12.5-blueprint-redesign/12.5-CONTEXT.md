## Implementation Decisions

### Voice capture pipeline

- **D-01 Provider = Deepgram Nova-3.** Lowest cost per minute (~$0.0043), lowest streaming latency (~200ms), strong NZ-accent support. New `DEEPGRAM_API_KEY` env var required; no existing dependency conflict. Billing relationship separate from the OpenAI/Anthropic accounts used by the SOP parser and verifier.
- **D-02 Transport = WebSocket streaming.** Audio streams from browser to a Next.js WebSocket endpoint (route TBD in planning â€” likely `/api/voice/ws/[sopId]` or a separate Node service proxied via Railway). Transcripts stream back as partial + final results. Delivers live transcript-as-spoken UX in the voice state machine (sketch: idle â†’ listening â†’ transcribing â†’ captured). Deepgram supports this natively; no polyfill needed. Open question for research: whether Next.js 16 App Router supports WebSocket routes directly, or if we need a sidecar Node server (Railway supports this) â€” research-phase to resolve.
- **D-03 Voice audio persistence.** Raw audio blob uploaded to Supabase Storage under `{org_id}/voice/{sop_id}/{uuid}.webm`. Transcript + metadata land in `sop_voice_notes` (D-04). Audio blob retained for 30 days, then purged via a new scheduled task (out of scope for 12.5, backlog).
- **D-04 Offline voice queue.** Voice capture while offline queues the audio blob + partial metadata in Dexie `voiceNotesQueue` store. Reconnect flushes the queue via the existing `sync-engine` pattern (Phase 12 precedent). Queued blobs hit Deepgram's REST endpoint for batch transcription (not streaming), since WebSocket requires live connection.

### Database schema

- **D-05 Single migration: `00021_blueprint_redesign.sql`.** Atomic bundle that adds:
  - `sop_voice_notes` â€” `(id uuid PK, organisation_id uuid, sop_id uuid, section_id uuid NULL, step_id uuid NULL, completion_id uuid NULL, block_type text CHECK IN ('measurement','note'), transcript text, audio_storage_path text, confidence numeric NULL, created_by uuid, created_at timestamptz)`. RLS: org-scoped via `auth.jwt()->>'organisation_id'`. Append-only for workers; admins can read all.
  - `escalation_reports` â€” `(id uuid PK, organisation_id uuid, sop_id uuid, section_id uuid, step_id uuid, completion_id uuid NULL, escalation_mode text CHECK IN ('alert','lock','form'), reason text, photos text[] NULL, measurements jsonb NULL, status text CHECK IN ('open','acknowledged','resolved'), submitted_by uuid, submitted_at timestamptz, acknowledged_by uuid NULL, acknowledged_at timestamptz NULL)`. RLS: org-scoped; workers see their own; supervisors see all in their org.
  - `walkthrough_progress` â€” `(sop_id uuid, user_id uuid, step_id uuid, completed_at timestamptz NULL, PRIMARY KEY (sop_id, user_id))`. Upserted on each step navigation. Supersedes any walkthrough state currently only in-memory. RLS: user sees their own.
  - `sops.flow_graph jsonb` â€” nullable ALTER to `sops` table. No backfill.
  - RLS policies + indexes in the same migration.
- **D-06 Voice note schema = one row per utterance.** Matches D-04. Keeps `sop_completions` append-only (Phase 4 immutability invariant). Enables full audit trail, easy querying for cmdk Ask AI context, and per-utterance confidence scores.

### CSS architecture

- **D-08 Tokens in `src/styles/blueprint-theme.css`**, imported from `src/app/layout.tsx`. Contains `:root` block with `--paper`, `--paper-2`, `--ink-*`, `--accent-*`, `--brand-yellow` carries over from existing Phase-12 token set. Also includes `.bg-grid` + `.bg-grid-dark` utility classes, typography classes (`.mono` / `.prose`), and all reusable primitives (`.pill`, `.tab`, `.evidence-btn`, `.measurement-box`, `.decision-btn`). Keeps `src/app/globals.css` focused on Phase-12 steel tokens. Tailwind extends via `tailwind.config.ts` `theme.extend.colors` referencing the CSS vars (see sketch-findings skill design-tokens.md for the exact config).
- **D-09 Theme scope via nested layout.** New `src/app/(protected)/sops/layout.tsx` sets `<body data-theme="paper">` (or applies it via a client component wrapping `{children}`). Paper/ink CSS is scoped under `body[data-theme="paper"]` selectors so admin routes (no `data-theme`) fall through to the existing steel-900 defaults. Zero CSS specificity fights; zero per-component Tailwind utility sprawl.
- **D-10 Public routes (`/`, `/login`, `/sign-up`).** These are NOT under the `(protected)/sops/` layout. Two options for applying paper/ink:
  - Option A (selected): a matching `src/app/(public)/layout.tsx` (or equivalent route-group layout) also sets `data-theme="paper"`.
  - Option B: apply `data-theme="paper"` at the root `src/app/layout.tsx` and let admin's `(protected)/admin/layout.tsx` override via `data-theme="steel"`.
  - **Decision: Option A.** Matches the cleaner scope pattern and avoids touching admin routes.

### Preview toggle on worker routes

- **D-11 Standalone `<WorkerPreviewToggle>` component.** Phase 12's Puck `viewports` prop only works inside a Puck context; worker tabs don't use Puck. New component lives at `src/components/sop/WorkerPreviewToggle.tsx`. Wraps its `{children}` in `<div style={{ maxWidth: viewport === 'mobile' ? 430 : '100%' }}>`. Toggle UI renders in the top chrome next to the tab nav. Preference persists to `localStorage` key `safestart.preview-viewport`. State lives in a small Zustand slice (`src/stores/preview.ts`) so every tab reads the same value without prop drilling.
- **D-12 Mobile preview is a clean column clamp, no phone frame.** The sketch had a skeuomorphic device frame in the admin builder; the worker-facing toggle omits it (cleaner, avoids duplicating Phase-12's admin aesthetic). Workers using the toggle want to see the content at 430px width, not a picture of a phone.

### Flow graph editor (inside Puck)

- **D-13 Custom Puck `FlowGraphField` with visual node-drag editor.** Registered on the SOP-level root (not on individual sections) at `src/lib/builder/flow-graph-field.tsx`. Type: Puck custom field (`type: 'custom'`, `render: ({value, onChange}) => <FlowGraphEditor value={value} onChange={onChange} />`). Editor is an SVG canvas with:
  - Palette on the left with 7 node types (Step, Measurement, Decision, Escalate, SignOff, Inspect, Zone) using the design-tokens accent colors
  - Drag to position on a 20px grid
  - Click a node edge + drag to another node to create an edge
  - Select edge â†’ choose edge kind (sequential / yes / no / escalate)
  - Right-side props panel when a node is selected, showing the node's Zod-backed fields (stepId linker, blockProps pass-through, label)
- **D-14 Flow graph Zod shape:**
  ```ts
  const FlowGraphSchema = z.object({
    version: z.literal(1),
    nodes: z.array(z.object({
      id: z.string().uuid(),
      type: z.enum(['step', 'measurement', 'decision', 'escalate', 'signoff', 'inspect', 'zone']),
      label: z.string().min(1).max(200),
      position: z.object({ x: z.number(), y: z.number() }),  // 20px-grid absolute
      stepId: z.string().uuid().optional(),
      blockProps: z.record(z.unknown()).optional(),  // pass-through for block-specific metadata
    })),
    edges: z.array(z.object({
      from: z.string().uuid(),
      to: z.string().uuid(),
      kind: z.enum(['sequential', 'yes', 'no', 'escalate']),
      label: z.string().max(60).optional(),
    })),
  })
  ```
  - Stored at `sops.flow_graph` (nullable JSONB). Admin edit saves via new `updateSopFlowGraph()` server action.
  - Exported from `src/lib/validators/flow-graph.ts` â€” available for `/api/schema` introspection.
- **D-15 Derivation function (`deriveFlowGraph(sop)`) at `src/lib/sop/flow-graph.ts`.** Walks `sop.sections â†’ steps â†’ layout_data.content[]` and produces the same `FlowGraphSchema` shape. Memoized via TanStack Query. Performance budget: â‰¤50ms for 500-step SOPs (SPEC constraint). Research-phase confirms SVG layout libraries that can help (elkjs, dagre).
- **D-16 Override precedence.** Flow tab checks `sop.flow_graph`: if non-null AND `FlowGraphSchema.safeParse(sop.flow_graph).success`, render as-is. Otherwise fall back to `deriveFlowGraph(sop)` and `console.warn('[flow] explicit graph invalid, using derived', parseResult.error)` once per page load (warn-once pattern from Phase 12's LayoutRenderer).

## Open Research Questions (for gsd-phase-researcher)

1. **Next.js 16 WebSocket support.** Does App Router support WebSocket routes out of the box, or do we need a sidecar Node service? If sidecar: is it deployable to Railway as a separate service, and how does it share the Supabase session cookie for auth?
2. **Deepgram integration idioms for NZ-accent.** Is there a `language: 'en-NZ'` tier or does it fall under `en-AU`? What's the correct auth header for WebSocket vs REST?
3. **SVG flow-graph layout library.** Do we hand-roll layout (absolute positions on a 20px grid set by the user) or pull in `elkjs` / `dagre`? What's the bundle-size cost?
4. **MediaRecorder cross-browser.** iOS Safari webm support is weak; may need `audio/mp4` fallback. Confirm Deepgram accepts both.
5. **Puck custom field state isolation.** When the admin edits the flow graph inside Puck, Puck's onChange fires per keystroke. Does our `FlowGraphEditor` debounce its own state, or rely on Puck's change batching?
