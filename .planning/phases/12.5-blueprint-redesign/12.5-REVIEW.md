# Phase 12.5: Code Review Report (Wave 5 — Flow Graph + Builder)

**Reviewed:** 2026-04-26T00:00:00Z

## Summary

Reviewed the Wave 5 gap-closure files covering the flow graph subsystem (validator, server action, builder field editor, FlowTab renderer), new block types (MeasurementBlock, VoiceNoteBlock, EscalationFormModal, VoiceCaptureControl), the CmdK / AI-ask pipeline, and supporting infrastructure (BuilderClient, puck-config, SOP types).

The overall architecture is sound. The two critical issues are: a role-gate in the flow-graph server action that reads from `user_metadata` (user-writable) instead of `app_metadata` (server-set only), allowing privilege escalation; and a client-side JWT decode in MeasurementBlock/VoiceNoteBlock that is both fragile and unnecessary given available context. Warnings cover: an undraned response body in CommandPalette, a missing `onCaptured` signal when offline voice transcripts are empty, an SVG `<defs>` marker declared after the edges that reference it, silent duplicate-edge creation in the flow editor, and a suppressed exhaustive-deps lint rule hiding a stale-closure bug in BuilderClient.

## Critical Issues

### CR-01: Role gate reads from user-writable `user_metadata` instead of `app_metadata`

**File:** `src/actions/flow-graph.ts:27-31`

**Issue:** The role check builds `rawRole` from `user.user_metadata?.user_role` (line 27-29). Supabase `user_metadata` is writable by authenticated users via `supabase.auth.updateUser()` — a user can set `user_metadata.user_role = 'admin'` from the client and bypass this gate. Line 30 also attempts to read `user.user_role` directly, which is not a real field on the `User` type; this cast always yields `undefined` and the logic falls through to `rawRole`. The authoritative, server-set-only source is `app_metadata.user_role`.

**Fix:**
```typescript
// Replace lines 27-31 with a single read from app_metadata (tamper-proof)
const role =
  (user as unknown as { app_metadata?: { user_role?: string } })
    .app_metadata?.user_role ?? ''
if (!['admin', 'safety_manager'].includes(role)) {
  return { error: 'Admin access required' }
}
```

Check whether other server actions in `src/actions/` use the same three-source role-read pattern — this is likely systemic.

### CR-02: Client-side JWT decode for `orgId` in MeasurementBlock and VoiceNoteBlock

**File:** `src/components/sop/blocks/MeasurementBlock.tsx:49-54`
**File:** `src/components/sop/blocks/VoiceNoteBlock.tsx:29-34`

**Issue:** Both blocks manually decode the JWT access token with `atob()` to extract `organisation_id`, then use that value to construct a Supabase Storage path (`${orgId}/voice/...`). Two problems:

1. JWT payloads are base64url-encoded (RFC 4648 §5) — they use `-`/`_` instead of `+`/`/` and omit `=` padding. `atob()` requires standard base64 with correct padding. The code adds back `+`/`/` correctly but the padding formula `'==='.slice((raw.length + 3) % 4)` is non-standard and will produce incorrect padding for some token lengths, causing `atob()` to throw a `DOMException` silently swallowed by the outer catch, leaving `orgId` undefined and the upload silently skipped.

2. Client-decoded JWT claims are not authenticated at this layer — if the token's `organisation_id` claim is spoofed (malformed JWT secret, dev misconfiguration), the storage path is written to the wrong org prefix. The correct source is `SopBlockContext` (server-fetched data) or a server action that resolves the org itself.

**Fix (preferred):** Add `organisationId` to `SopBlockContext` and read it from there:
```typescript
// In handleCaptured — replace the JWT decode block with:
const orgId = ctx.organisationId  // added to SopBlockContext
if (!orgId) return
```

If keeping the client-side upload, at minimum fix the base64url decoding:
```typescript
const raw = session?.access_token?.split('.')[1]
  ?.replace(/-/g, '+').replace(/_/g, '/')
const padded = raw ? raw + '='.repeat((4 - (raw.length % 4)) % 4) : null
const claims = padded
  ? (JSON.parse(atob(padded)) as { organisation_id?: string })
  : {}
```

## Warnings

### WR-01: Response body not consumed after non-OK fetch in CommandPalette — connection leak

**File:** `src/components/sop/CommandPalette.tsx:35`

**Issue:** When `res.ok` is false, the code sets an error state and returns without draining `res.body`. In HTTP/2 and keep-alive connections, an unconsumed body holds the stream open until the browser times it out. The command palette can be opened and closed rapidly, exhausting the browser connection pool.

**Fix:**
```typescript
if (!res.ok || !res.body) {
  await res.body?.cancel()   // drain / release the connection
  setAiError('Request failed')
  setAiLoading(false)
  return
}
```

### WR-02: Empty offline voice transcript stored as `undefined` — lost on sync

**File:** `src/components/sop/VoiceCaptureControl.tsx:96-97`

**Issue:** When offline, the Dexie queue row is written with `transcript: result.transcript || undefined`. If Deepgram returns an empty final transcript (e.g., recording too short, silence), `result.transcript` is `''`, which is falsy, so the row stores `transcript: undefined`. On reconnect the sync engine has no text to surface and the voice note is persisted with a blank transcript — the offline data is silently degraded.

**Fix:** Store empty string rather than dropping the field:
```typescript
transcript: result.transcript ?? '',
```

### WR-03: SVG `<defs>` / `<marker>` declared after the edges that reference it

**File:** `src/lib/builder/flow-graph-field.tsx:305-309`

**Issue:** The `<defs>` block containing the `#arrow` marker is rendered after the `{edges.map(...)}` JSX. SVG painting order means edges render before the marker definition exists in the document. The SVG spec states that `<defs>` content should be declared before use; while most Chromium versions resolve forward references within the same SVG, WebKit and some SVG-to-PDF/print renderers do not, so arrowheads will be missing in those environments.

**Fix:** Move `<defs>` to immediately after the opening `<svg>` tag, before the edges block.

### WR-04: Duplicate edges silently created in flow editor

**File:** `src/lib/builder/flow-graph-field.tsx:91-98`

**Issue:** Edge creation in edge-mode does not check for an existing edge between the same `from` and `to` pair. Clicking source → target multiple times adds multiple overlapping `sequential` edges. They are invisible on the canvas (stacked), inflate the saved graph, and survive round-trip through `FlowGraphSchema.safeParse` (the schema validates but does not deduplicate).

**Fix:**
```typescript
} else if (edgeSrcId !== nodeId) {
  setEdges((es) => {
    if (es.some((e) => e.from === edgeSrcId && e.to === nodeId)) return es
    return [...es, { from: edgeSrcId, to: nodeId, kind: 'sequential' as const }]
  })
  setEdgeSrcId(null)
  setEdgeMode(false)
}
```

### WR-05: Suppressed exhaustive-deps hides stale `sections` closure in overwrite toast

**File:** `src/app/(protected)/admin/sops/builder/[sopId]/BuilderClient.tsx:106`

**Issue:** The `useEffect` on lines 96-107 maps `lastSyncResult.overwrittenByServer` IDs to section titles using `sections`, but `sections` is excluded from the dependency array via the eslint-disable comment on line 106. If `sections` changes (section added or renamed) between renders, the toast will display stale titles or fall back to the ID slice. This is a real stale-closure bug masked by the lint suppression.

**Fix:** Add `sections` to the dependency array — the effect is cheap and re-running on section changes is correct:
```typescript
  }, [lastSyncResult, sections])
```

### IN-03: `deriveFlowGraph` performs a `sections.find()` inside a `steps` loop — O(S×T) scan

**File:** `src/lib/sop/flow-graph.ts:26-38`

**Issue:** For each step in `allSteps`, a `sop_sections.find()` scans all sections to locate the parent. For typical SOP sizes (10-20 sections, 100-200 steps) this is negligible. Noted for awareness if SOPs grow large.

**Suggestion (low priority):** Build a `Map<stepId, SopSection>` before the loop to reduce to O(S + T).

### IN-04: `FlowTab` `useMemo` dependency on `sop.updated_at` may not capture step-level mutations

**File:** `src/components/sop/tabs/FlowTab.tsx:137`

**Issue:** `useMemo(() => deriveFlowGraph(sop), [sop.id, sop.updated_at])` correctly memoizes on stable scalar fields. However, if a step is mutated client-side (optimistic update) without refreshing `sop.updated_at`, the derived graph will be stale until the next server refetch. Not a current bug given the data flow (server fetch → React Query cache), but worth documenting before optimistic step mutations are added.

**Suggestion:** Add an inline comment noting the `updated_at` dependency requirement so future optimistic-update work doesn't silently break the memo.
