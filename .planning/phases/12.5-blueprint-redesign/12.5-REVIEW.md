# Phase 12.5: Code Review Report

**Reviewed:** 2026-04-26T10:35:42Z
**Depth:** standard
**Files Reviewed:** 20
**Status:** issues_found

## Summary

Reviewed 20 source files covering the Phase 12.5 blueprint redesign: voice capture pipeline, offline queue, escalation server actions, walkthrough state, and the new immersive mode. The voice token endpoint correctly gates the Deepgram API key server-side. Server actions use Supabase `getUser()` (not just `getSession()`) for primary auth, which is correct. Zod validation is consistently applied at all server action entry points.

Three critical issues were found:

1. The JWT claim extraction in `voice-notes.ts` and `escalation.ts` uses a manual base64-decode of the access token payload instead of the Supabase SDK's built-in `session.user.app_metadata` / `user_metadata`. The current approach silently produces an empty `claims` object when `session` is `null` (possible between token refresh cycles), bypassing the org isolation guard.

2. `VoiceCaptureControl` calls `useCallback` inside the function body after an early conditional `return`, violating the Rules of Hooks. This will throw a runtime error in React 19 strict mode.

3. The offline voice-queue flush sends `atob()` to decode the JWT in a client context. `atob` on Node.js is available in Next.js edge/server-side but the `voice-queue.ts` module is marked `'use client'` — meaning this runs in the browser where `atob` exists, but the base64url padding issue (no `=` padding in JWTs) means the decode will throw a `DOMException: Failed to decode` on some inputs, silently leaving voice notes un-synced.

Six warnings cover: resource leak on mic stream when `getUserMedia` succeeds but WebSocket `onopen` never fires; missing `onClose` guard in `EscalationFormModal`; `handleStart`/`handleStop` in `VoiceCaptureControl` receiving stale `props` reference via closure; the `walkthrough-progress` action not validating org scope (any authenticated user can write any SOP's progress); the `EscalateBlock` form-mode `onSubmit` not setting `status('error')` on failure; and a double render of `ImmersiveStepCard` in `WalkthroughTab` for all viewport sizes simultaneously.

## Critical Issues

### CR-01: JWT org-claim extraction silently returns `{}` when session is null

**File:** `src/actions/voice-notes.ts:29-37` (same pattern in `src/actions/escalation.ts:17-26`)

**Issue:** `supabase.auth.getSession()` can return `{ data: { session: null } }` in the brief window between an access token expiry and a background refresh. When that happens, the ternary falls to `{}` and `claims.organisation_id` is `undefined`, causing the action to return `{ error: 'Missing organisation_id claim' }` — but, more critically, the pattern means org isolation depends entirely on a manually decoded JWT claim rather than a server-verified source. If an attacker crafts a JWT with a spoofed `organisation_id` in the payload (possible with a compromised Supabase JWT secret or during development with `SUPABASE_JWT_SECRET` misconfigured), the `organisation_id` they inject bypasses RLS-at-the-app-layer isolation.

The correct pattern for Supabase Server Actions is to read the org from `user.app_metadata` (server-verified claim set by the Supabase Auth hook, not user-modifiable) or to perform a DB lookup scoped by `user.id`.

**Fix:**
```typescript
// In both voice-notes.ts and escalation.ts — replace the manual JWT decode:

// BEFORE (brittle):
const { data: { session } } = await supabase.auth.getSession()
const claims = session?.access_token
  ? (JSON.parse(Buffer.from(session.access_token.split('.')[1], 'base64').toString()) as { organisation_id?: string })
  : {}
const orgId = claims.organisation_id
if (!orgId) return { error: 'Missing organisation_id claim' }

// AFTER (safe — reads from server-verified app_metadata):
const orgId = user.app_metadata?.organisation_id as string | undefined
if (!orgId) return { error: 'Missing organisation_id claim' }
```

If `organisation_id` lives in a custom JWT claim (Supabase Hook) rather than `app_metadata`, use a DB lookup instead:
```typescript
const { data: member } = await supabase
  .from('organisation_members')
  .select('organisation_id')
  .eq('user_id', user.id)
  .single()
if (!member) return { error: 'Not a member of any organisation' }
const orgId = member.organisation_id
```

### CR-02: React Rules of Hooks violation — `useCallback` after conditional return

**File:** `src/components/sop/VoiceCaptureControl.tsx:55-77`

**Issue:** The component calls `isVoiceCaptureSupported()` at line 55, then conditionally `return`s JSX at line 56-61 if not supported. The `useCallback` hooks for `handleStart` (line 64) and `handleStop` (line 79) are declared after this conditional return. React requires all hooks to be called unconditionally and in the same order on every render. This will throw `Error: Rendered fewer hooks than expected` in development and may silently corrupt hook state in production builds. React 19 strict mode enforces this more aggressively.

**Fix:** Move `handleStart` and `handleStop` above the early return, or restructure to avoid the conditional return entirely:

```typescript
export function VoiceCaptureControl(props: VoiceCaptureControlProps) {
  const [state, dispatch] = useReducer(reducer, { kind: 'idle' })
  const { start, stop } = useDeepgramWebSocket()
  const isOnline = useNetworkStore((s) => s.isOnline)

  // All useCallback hooks BEFORE any early return
  const handleStart = useCallback(async () => {
    // ... existing body
  }, [start, props])

  const handleStop = useCallback(async () => {
    // ... existing body
  }, [stop, isOnline, props])

  // Now safe to do early return
  const supported = typeof window !== 'undefined' ? isVoiceCaptureSupported() : true
  if (!supported) {
    return (
      <span className="text-xs text-[var(--ink-500)]">
        Voice capture requires iOS 14.3+ or desktop Chrome/Firefox/Safari
      </span>
    )
  }

  return ( /* ... rest of JSX */ )
}
```

### CR-03: base64url JWT decode without padding strips — throws `DOMException` on some tokens

**File:** `src/lib/offline/voice-queue.ts:58-64`

**Issue:** JWT segments are base64url-encoded (RFC 4648 §5), which omits `=` padding and uses `-`/`_` instead of `+`/`/`. `atob()` requires standard base64 with correct padding. When the JWT payload length mod 4 is non-zero (common), `atob(session.access_token.split('.')[1])` throws `DOMException: Failed to execute 'atob'`. This silently causes the entire `flushVoiceNoteQueue` loop iteration to throw, leaving the voice note marked `dirty` forever — no sync, no error surfaced to the user.

The same bug exists server-side in `voice-notes.ts` and `escalation.ts` (lines 30-35 and 19-25 respectively) but uses `Buffer.from(..., 'base64')` which is more tolerant of base64url. Still, the correct fix for all three files is CR-01's approach (avoid manual JWT decode entirely).

**Fix for `voice-queue.ts` specifically:**
```typescript
// Replace:
const claims = session?.access_token
  ? (JSON.parse(atob(session.access_token.split('.')[1])) as { organisation_id?: string })
  : {}

// With (adds padding + replaces base64url chars):
function decodeJwtPayload(token: string): Record<string, unknown> {
  try {
    const segment = token.split('.')[1].replace(/-/g, '+').replace(/_/g, '/')
    const padded = segment + '='.repeat((4 - (segment.length % 4)) % 4)
    return JSON.parse(atob(padded)) as Record<string, unknown>
  } catch {
    return {}
  }
}

const claims = session?.access_token ? decodeJwtPayload(session.access_token) : {}
```

However, the preferred fix remains CR-01: source the `organisation_id` from `user.app_metadata` or a DB lookup rather than parsing it from the JWT in the client.

## Warnings

### WR-01: Mic stream not released when WebSocket never opens

**File:** `src/lib/voice/deepgram-stream.ts:51-82`

**Issue:** `navigator.mediaDevices.getUserMedia({ audio: true })` is called at line 51 before the WebSocket has confirmed it is open. If the WebSocket `onerror` fires before `onopen` (e.g. Deepgram returns 401 because the ephemeral token is already expired), `recorder.start()` is never called, but `stream.getTracks()` are never stopped either. The browser's microphone indicator stays active until the tab is closed or the component unmounts. The `onError` callback fires (line 83), but the mic stream tracks are not included in that code path.

**Fix:** Stop mic tracks in `ws.onerror` and also in the `ws.onclose` handler when the recorder was never started:
```typescript
let recorderStarted = false

ws.onopen = () => {
  recorder.start(250)
  recorderStarted = true
  keepAlive = setInterval(...)
}

ws.onerror = () => {
  if (!recorderStarted) {
    for (const t of stream.getTracks()) t.stop()
  }
  errorCb(new Error('WebSocket error'))
}
```

### WR-02: `handleStart`/`handleStop` capture stale `props` reference

**File:** `src/components/sop/VoiceCaptureControl.tsx:64-108`

**Issue:** Both `useCallback` hooks declare `props` in their dependency arrays (lines 77, 108). In React, the entire `props` object reference changes on every render, so these callbacks are re-created on every render — making `useCallback` a no-op for memoization. Worse, because `props` is captured as a whole-object reference, any stale closure on `handleStop` retains the full previous `props` object including `sopId`, `stepId`, etc. If the parent re-renders with new context IDs between `handleStart` and `handleStop`, the offline queue entry will be written with the props at the time `handleStart` was called (likely still correct), but this is fragile.

**Fix:** Destructure the specific values used in each callback dependency array:
```typescript
const { sopId, sectionId, stepId, completionId, language, target, onTranscript } = props

const handleStart = useCallback(async () => {
  // use destructured values
}, [start, language, target, onTranscript])

const handleStop = useCallback(async () => {
  // use destructured values
}, [stop, isOnline, sopId, sectionId, stepId, completionId, language, target])
```

### WR-03: `walkthrough-progress` action writes progress for any SOP — no org scope check

**File:** `src/actions/walkthrough-progress.ts:10-32`

**Issue:** The `upsertWalkthroughProgress` action validates that `sopId` is a UUID and that the user is authenticated, but it does not verify that the SOP belongs to the user's organisation. An authenticated user from Org A who obtains a valid UUID for an SOP in Org B can write a `walkthrough_progress` row for that SOP, poisoning progress tracking for Org B's workers. This is likely mitigated by RLS on the `walkthrough_progress` table if `organisation_id` is a column there, but the application layer provides no defence in depth.

**Fix:** Add an org-scope check before the upsert:
```typescript
// Verify sop belongs to user's org before writing progress
const { data: sop, error: sopErr } = await supabase
  .from('sops')
  .select('id')
  .eq('id', parsed.data.sopId)
  .maybeSingle()
if (sopErr || !sop) return { error: 'SOP not found or access denied' }
```

RLS on `sops` will enforce org scope, so the select acts as the check.

### WR-04: `EscalateBlock` form-mode `onSubmit` swallows errors — `status` stays `'idle'`

**File:** `src/components/sop/blocks/EscalateBlock.tsx:122-136`

**Issue:** The `onSubmit` handler passed to `EscalationFormModal` (line 125-133) does not wrap `submitEscalationReport` in a try/catch. If the server action throws or returns an error, the `await` resolves/rejects silently — `EscalationFormModal` catches it in its `finally` block (line 59) so `busy` resets, but neither `setModalOpen(false)` nor `setStatus('sent')` is called. The modal remains open indefinitely. More importantly, `setStatus('error')` is never called in the form path — only in the `alert`/`lock` paths (line 59).

**Fix:**
```typescript
onSubmit={async (payload) => {
  const result = await submitEscalationReport({
    sopId: ctx.sopId,
    sectionId: ctx.sectionId,
    stepId: ctx.stepId,
    completionId: ctx.completionId,
    ...payload,
  })
  if ('error' in result) {
    setModalOpen(false)
    setStatus('error')
    return
  }
  setModalOpen(false)
  setStatus('sent')
}}
```

### WR-05: Double render of `ImmersiveStepCard` — both DOM nodes active simultaneously

**File:** `src/components/sop/tabs/WalkthroughTab.tsx:34-68`

**Issue:** `WalkthroughTab` renders `ImmersiveStepCard` twice unconditionally — once inside `walkthrough-list-only-above-430` (lines 35-37) when in immersive mode, and again inside `immersive-only-below-430` (lines 65-67). When the viewport is above 430px in immersive mode, both instances are in the DOM at the same time. Each calls `useSearchParams()` and `useWalkthroughStore()` independently. `onStepChange` is fired twice per navigation gesture (once per instance if both are interactive). On mobile the list-mode branch also renders a list inside `walkthrough-list-only-above-430` that is purely CSS-hidden above 430px but still subscribes to the store.

This is a logic error rather than just a style issue: two `ImmersiveStepCard` nodes race on step navigation, and the `useWalkthroughStore` subscription runs twice, making state reads redundant and potentially triggering double-calls to `upsertWalkthroughProgress`.

**Fix:** Gate the second render with a `useMediaQuery` hook or pass a prop to skip rendering when CSS would hide it, rather than relying on CSS alone for DOM presence:
```typescript
// Quick structural fix — only render the "below 430" instance when mode isn't immersive:
{mode !== 'immersive' && (
  <div className="immersive-only-below-430">
    <ImmersiveStepCard sop={sop} onStepChange={handleStepChange} />
  </div>
)}
```

### WR-06: Deepgram REST auth in offline flush uses `Bearer` but streaming WebSocket uses `token` subprotocol — encoding mismatch risk

**File:** `src/lib/offline/voice-queue.ts:31-38`

**Issue:** The offline flush uses `Authorization: Bearer ${access_token}` (line 34) against the Deepgram REST endpoint. The live streaming path in `deepgram-stream.ts` uses the WebSocket `Sec-WebSocket-Protocol` subprotocol with `['token', access_token]` (line 45-48). Deepgram's ephemeral tokens minted via `/v1/auth/grant` are scoped for the streaming API. Whether they are also valid for the Deepgram REST transcription endpoint (`/v1/listen` POST) depends on Deepgram's token grant policy, which is undocumented for ephemeral tokens. If the REST endpoint rejects ephemeral tokens, the `Deepgram REST ${dgRes.status}` error at line 39 causes all offline voice blobs to fail permanently (they stay dirty but the transcript from the capture-time pass is lost after the first retry throws).

**Fix:** This should be tested explicitly. If Deepgram REST does not accept ephemeral tokens, the offline flush path needs to call the server-side `/api/voice/token` endpoint (as it already does at line 18-20) but may need a different token grant type, or fall back to sending the pre-captured `note.transcript` without re-transcription when offline:
```typescript
// Defensive fallback: if Deepgram REST fails, use the transcript already captured
// at recording time rather than losing the note permanently
const transcript = (dgRes.ok ? alt?.transcript : undefined) ?? note.transcript ?? ''
```

### IN-01: `SopBlockContext` has no consumer guard hook — missing context is silent in blocks

**File:** `src/components/sop/SopBlockContext.tsx:1-21`

**Issue:** `SopBlockContext` is initialized with `null` and blocks use `useContext(SopBlockContext)` directly. The `MeasurementBlock` and `VoiceNoteBlock` handle the null case gracefully (render a disabled fallback). `EscalateBlock` also handles it (shows a hint at line 117-120) but only after the button is rendered. There is no exported `useSopBlockContext()` hook that throws a descriptive error in development, which means future blocks could silently receive `null` and fail at runtime.

**Fix:** Export a typed hook alongside the context:
```typescript
export function useSopBlockContext(): SopBlockCtx | null {
  return useContext(SopBlockContext)
}

// For blocks that require the context (not optional):
export function useRequiredSopBlockContext(blockName: string): SopBlockCtx {
  const ctx = useContext(SopBlockContext)
  if (!ctx && process.env.NODE_ENV === 'development') {
    throw new Error(`${blockName} must be rendered inside <SopBlockProvider>`)
  }
  return ctx!
}
```

### IN-02: `VoiceNoteBlock` prop `maxDurationSec` is accepted but never enforced

**File:** `src/components/sop/blocks/VoiceNoteBlock.tsx:13-17`

**Issue:** `maxDurationSec` is validated by Zod schema (min 5, max 300), passed to `VoiceCaptureControl` display label, and stored in `data-max-duration` — but `VoiceCaptureControl` accepts no `maxDurationSec` prop and has no recording timeout. A user can record indefinitely. This is a missing feature not a bug, but the schema implies enforcement.

// Inside handleStart, after dispatch({ type: 'START' }):
if (props.maxDurationSec) {
  setTimeout(() => void handleStop(), props.maxDurationSec * 1000)
}

### IN-03: `walkthroughMode.ts` store initializes with `readInitial()` at module load time — SSR hydration mismatch

**File:** `src/stores/walkthroughMode.ts:7-16`

**Issue:** `readInitial()` is called at `create<State>()` time (line 23). On the server, `typeof window === 'undefined'` returns `'list'`. On the client, it reads `localStorage`. This means the initial Zustand snapshot differs between SSR and CSR, producing a React hydration mismatch warning for any component that reads `mode` during SSR (e.g., `WalkthroughTab`).

**Fix:** Use Zustand's `createWithEqualityFn` with a `skipHydration: true` pattern or initialize to `'list'` unconditionally and hydrate in a `useEffect`:
```typescript
export const useWalkthroughModeStore = create<State>((set) => ({
  mode: 'list', // safe SSR default
  setMode: (m) => { ... },
}))

// In a client component (e.g., WalkthroughTab):
useEffect(() => {
  useWalkthroughModeStore.getState().setMode(readInitial())
}, [])
```
