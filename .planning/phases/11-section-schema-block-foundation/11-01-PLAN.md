    Append RLS enable + policies to the migration (Step 7). Policy convention:
    - **SELECT**: allow where `organisation_id is null OR organisation_id = current_organisation_id()` (includes globals + own-org).
    - **INSERT/UPDATE/DELETE** from authenticated role: allow ONLY where `organisation_id = current_organisation_id() AND current_user_role() in ('admin','safety_manager')`. This FORBIDS authenticated writes to globals (organisation_id IS NULL) — globals are service_role only, and seeds ship via this migration.

    ```sql
    -- ============================================================
    -- Step 7: RLS
    -- ============================================================
    alter table public.section_kinds enable row level security;
    alter table public.blocks enable row level security;
    alter table public.block_versions enable row level security;
    alter table public.sop_section_blocks enable row level security;

    -- ---------- section_kinds ----------
    create policy "section_kinds_read_global_plus_org"
      on public.section_kinds for select to authenticated
      using (organisation_id is null or organisation_id = public.current_organisation_id());

    create policy "section_kinds_admin_insert_own_org"
      on public.section_kinds for insert to authenticated
      with check (
        organisation_id = public.current_organisation_id()
        and public.current_user_role() in ('admin','safety_manager')
      );

    create policy "section_kinds_admin_update_own_org"
      on public.section_kinds for update to authenticated
      using (
        organisation_id = public.current_organisation_id()
        and public.current_user_role() in ('admin','safety_manager')
      )
      with check (
        organisation_id = public.current_organisation_id()
        and public.current_user_role() in ('admin','safety_manager')
      );

    create policy "section_kinds_admin_delete_own_org"
      on public.section_kinds for delete to authenticated
      using (
        organisation_id = public.current_organisation_id()
        and public.current_user_role() in ('admin','safety_manager')

    -- ---------- blocks ----------

      on public.blocks for select to authenticated

    create policy "blocks_admin_insert_own_org"
      on public.blocks for insert to authenticated

        organisation_id = public.current_organisation_id()

    create policy "blocks_admin_update_own_org"
      on public.blocks for update to authenticated

        and public.current_user_role() in ('admin','safety_manager')

        and public.current_user_role() in ('admin','safety_manager')
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated role → RLS | Admin/safety_manager can insert/update/delete their own org's section_kinds, blocks, block_versions, sop_section_blocks. No authenticated role can touch `organisation_id IS NULL` rows. |
| service_role → global rows | Only service_role (migrations, ops scripts) writes global section_kinds and global blocks. The seed INSERT in this migration runs as superuser during `supabase db push`. |
| Dexie / worker client → snapshot_content | Workers read `sop_section_blocks.snapshot_content` via joined SELECTs; they never directly write to blocks/block_versions (RLS forbids it because worker role is not admin). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01-01 | Elevation of Privilege | section_kinds / blocks RLS | mitigate | Every INSERT/UPDATE/DELETE policy explicitly requires `organisation_id = public.current_organisation_id()` (non-null equality) AND `public.current_user_role() in ('admin','safety_manager')`. Authenticated role cannot write `organisation_id IS NULL` rows — only service_role (migration context) seeds globals. Task-3 verify step greps for any auth-role insert permitting NULL org_id and fails if present. |
| T-11-01-02 | Tampering | block_versions audit trail | mitigate | No UPDATE or DELETE policy is created for `block_versions` under the authenticated role. Only INSERT (new version rows) and SELECT are allowed. Version history is append-only. |
| T-11-01-03 | Information Disclosure | cross-tenant read of blocks | mitigate | `blocks_read_global_plus_org` uses `organisation_id is null OR organisation_id = public.current_organisation_id()`. `block_versions_read_via_blocks` and `ssb_read_own_org` both scope via EXISTS joins that filter by `current_organisation_id()`. Tests in Phase 11 regression (11-02) + `rls-isolation.test.ts` confirm no cross-tenant leak. |
| T-11-01-04 | Information Disclosure | section_kind leak across orgs | mitigate | `section_kinds` read policy unions globals (NULL org_id) + own-org rows. Different orgs cannot see each other's custom kinds. |
| T-11-01-05 | Tampering | snapshot_content drift | accept (documented) | `snapshot_content` is cached at insertion. The `overridden_at` column signals intentional admin edits. Follow-latest updates are handled by an application-layer job in a later phase, never by direct SQL; any out-of-band write is audit-traceable via updated_at. |
| T-11-01-06 | Denial of Service | large JSONB payloads | accept | Postgres default jsonb max is well above SOP block sizes (hundreds of KB worst case). No explicit limit enforced; application layer (Phase 13) imposes a 64KB soft cap on `block_versions.content`. |
| T-11-01-07 | Injection | raw SQL inside migration | mitigate | The migration is a plain SQL file executed via `supabase db push`; no dynamic input. Seed values are string literals only. No `execute format(...)` dynamic SQL. |
| T-11-01-08 | Information Disclosure | secrets in migration | mitigate | Migration contains no passwords, tokens, or PII. Seed content is public metadata (icon names, color tokens, lucide-react icon identifiers). |
| T-11-01-09 | Spoofing | cross-tenant write via nullable FK | mitigate | The unique index on `(coalesce(org_id, sentinel), slug)` uses a fixed UUID sentinel so no org can create a row that shadows a global kind — (`NULL`,'hazards') and (`<org>`,'hazards') are distinct. Application queries must always filter by org to decide which one applies. |
