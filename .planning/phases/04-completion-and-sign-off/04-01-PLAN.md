    export interface LocalCompletion {

    **2. Create src/stores/completionStore.ts (Zustand, persisted to Dexie):**

    This is a SEPARATE store from walkthrough.ts (which remains memory-only per Phase 3 safety decision).

    ```typescript
    interface CompletionStoreState {
      // Active in-progress completions: sopId -> LocalCompletion
      activeCompletions: Record<string, LocalCompletion>
      startCompletion: (sopId: string, sopVersion: number) => Promise<void>
      markStepCompleted: (sopId: string, stepId: string) => Promise<void>
      getActiveCompletion: (sopId: string) => LocalCompletion | null
      clearCompletion: (sopId: string) => Promise<void>
      restoreFromDexie: (sopId: string) => Promise<LocalCompletion | null>
    }
    ```

    Key behaviors:
    - `startCompletion`: Creates a new LocalCompletion with `crypto.randomUUID()` as localId, writes to Dexie immediately
    - `markStepCompleted`: Updates stepCompletions record (stepId -> Date.now()), writes updated record to Dexie
    - `restoreFromDexie`: On walkthrough page load, checks if an in_progress completion exists for this sopId in Dexie. If found, restores it into the Zustand store (D-02 resume support)
    - `clearCompletion`: Removes from both Zustand state and Dexie (called after successful submit)
    - Every mutation writes to Dexie `db.completions.put(...)` for durability

    **3. Create src/lib/offline/photo-compress.ts (Canvas API compression):**

    Export `compressPhoto(file: File, maxDimension?: number, targetBytes?: number): Promise<Blob>`
    - Default maxDimension = 1200, targetBytes = 200_000
    - Load image via `new Image()` + `URL.createObjectURL(file)`
    - Scale down if exceeds maxDimension
    - Binary search quality from 0.1 to 0.9 via `canvas.toBlob(callback, 'image/jpeg', quality)`
    - Returns the compressed Blob
    - Follows the exact pattern from 04-RESEARCH.md Pattern 2

    **4. Extend src/lib/offline/sync-engine.ts with two new exports:**

    `flushCompletions(supabase: AnySupabaseClient)`:
    - Reads all Dexie completions where `status = 'submitted'` (NOT in_progress)
    - For each: reads associated photoQueue entries, collects storage paths of uploaded photos
    - Calls the submitCompletion server action (imported from @/actions/completions)
    - On success (or idempotent conflict): marks Dexie completion as synced by deleting it
    - On failure: leaves in Dexie for retry

    `flushPhotoQueue(supabase: AnySupabaseClient)`:
    - Reads all Dexie photoQueue entries where `uploaded === false` (Dexie boolean false = 0)
    - For each: calls getPhotoUploadUrl server action, PUTs blob to presigned URL, updates Dexie record with `uploaded: true` and `storagePath`
    - Returns `{ uploaded: number, errors: string[] }`

    **5. Create src/actions/completions.ts with three server actions:**

    a. `submitCompletion(rawInput: unknown)`:
    - Zod schema: `{ localId: uuid, sopId: uuid, sopVersion: int, contentHash: string(64), stepData: Record<string, number>, photoStoragePaths: array of { localId: uuid, stepId: uuid, storagePath: string, contentType: string } }`
    - Auth check via createClient().auth.getUser()
    - Extract orgId from JWT custom claims (user.app_metadata.organisation_id)
    - Use createAdminClient() to INSERT into sop_completions with `id: parsed.data.localId` (client UUID as PK for idempotent retry)
    - submitted_at is OMITTED — uses DB DEFAULT now() (COMP-01 server-side timestamp)
    - On conflict (23505): return success (idempotent, already submitted)
    - INSERT completion_photos for each photo storage path
    - Return `{ success: true, completionId }` or `{ success: false, error }`

    b. `signOffCompletion(rawInput: unknown)`:
    - Zod schema: `{ completionId: uuid, decision: 'approved' | 'rejected', reason?: string }`
    - Auth: verify caller is supervisor or safety_manager (JWT claim check)
    - For 'rejected': verify reason is non-empty (min 10 chars)
    - For supervisor: verify the completion's worker_id is in supervisor_assignments for this supervisor
    - For safety_manager: no extra check (sees all org-wide, D-14)
